## 一、问题清单

### 问题 1：`road_rage_voice_record` 表分片键与表结构不匹配

- **所在位置**：§3.1.5（分库分表策略） vs §3.1.1（AR-05 RoadRageVoiceRecord 表结构）
- **严重程度**：**严重（阻塞实现）**
- **问题描述**：§3.1.5 将 `road_rage_voice_record` 列在"按 VehicleId 分片"表中，§3.5.2 的分片表配置中亦将其列为按 VehicleId 分片。但 §3.1.1 中 RoadRageVoiceRecord 表字段仅有 `record_id`、`alert_id`、`trip_id`、`driver_id` 等，**不包含 `vehicle_id` 列**。若 ShardingSphere-JDBC 按 `hash(vehicle_id)` 路由，该表缺少分片键列，无法实施分片路由。
- **改进建议**：为该表增加冗余 `vehicle_id` VARCHAR(64) NOT NULL 列，并在 §3.1.1 表结构中补充；或将该表改为按 `driver_id` 哈希分片并调整 §3.1.5 和 §3.5.2 的分片范围说明。

### 问题 2：分片策略使用取模运算而非一致性哈希，与扩容目标矛盾

- **所在位置**：§3.5.2（金仓数据库连接与分片配置） vs §3.1.5（分库分表策略）
- **严重程度**：**严重（阻塞实现）**
- **问题描述**：§3.5.2 明确写为 `hash(vehicle_id) % 4`，并以括号标注"一致性哈希路由"。取模哈希（`% N`）与一致性哈希是两种不同的路由策略——取模哈希在变更分片数量 N 时需全量数据重分布，一致性哈希通过虚拟节点和哈希环使扩容时仅影响少数分片。§3.1.5 同时声称"按数据增长动态扩容"，取模方案与动态扩容目标直接矛盾。需求文档 §五.2 明确要求"按 VehicleId 一致性哈希路由"。
- **改进建议**：将路由算法改为 ShardingSphere-JDBC 的一致性哈希分片算法（去掉 `% 4` 取模运算），或明确声明本期固定 4 片不扩容、扩容需接受全量重分布，并给出迁移方案。

### 问题 3：`EmergencyRescueService` 的部署位置标注错误

- **所在位置**：§3.3.6（事件消费者注册配置表）
- **严重程度**：**中等（逻辑矛盾）**
- **问题描述**：§3.3.6 中 `EmergencyActivatedEvent` 的消费方第一项为"EmergencyRescueService（边缘）"。但领域层 OOD（`docs/ood_domain.md` DS-12）明确 EmergencyRescueService 的职责包含"向救援中心/120 发送精准定位"、"云端授权开启车门锁"、校验 RescueAuthorizationToken 并调用外部 HTTP/SMN 接口——这些操作依赖云端网络连通性，无法在边缘侧执行。该服务应是云端消费者。
- **改进建议**：将该项标注改为"EmergencyRescueService（云端）"，与领域层 DS-12 的实际运行环境保持一致。

### 问题 4：`SafetyAlertEvent` 主表存在与 CQRS 读模型投影冗余的复合索引

- **所在位置**：§3.1.2（E-01 SafetyAlertEvent 索引设计）
- **严重程度**：**中等（资源浪费）**
- **问题描述**：设计在 SafetyAlertEvent **主表**上建立了三条索引：(a) `(driver_id, occurred_at DESC, alert_type, risk_level)` 复合索引，(b) `(vehicle_id, occurred_at DESC)` 索引，(c) `(trip_id)` FK 索引。但同节"查询策略"明确声明"读侧：跨行程/跨驾驶员的告警历史查询…均通过 CQRS 投影表（`alert_projection`）完成，不穿透 Trip 聚合根加载"。这意味着索引 (a) 和 (b) 所服务的读查询场景完全由 `alert_projection` 投影表覆盖。写侧仅通过 `(trip_id)` FK 索引加载 Trip 聚合内的告警列表。三个索引中有两个是冗余的读侧索引，且 4 字段复合 B-tree 索引 (a) 会显著拖累每次告警 INSERT 的写性能。
- **改进建议**：主表仅保留 `(trip_id)` FK 索引（写侧唯一依赖的索引）。将 `(driver_id, occurred_at DESC, alert_type, risk_level)` 和 `(vehicle_id, occurred_at DESC)` 索引限定于 `alert_projection` 投影表（已在 §3.2.3 定义），与 CQRS 读模型设计保持索引职责边界一致。

### 问题 5：`EdgeSessionContext` 被多处引用但基础设施层未定义其管理方式

- **所在位置**：§3.4.1、§3.4.2
- **严重程度**：**中等（关键概念缺失）**
- **问题描述**：§3.4.1 称"一个 EdgeSessionContext 对应一个 VehicleStateBuffer 实例"；§3.4.1 称"边缘侧随行程开始创建（点火时初始化），随行程结束销毁"。领域层 OOD（`docs/ood_domain.md` §6.2）已定义 EdgeSessionContext 的概念模型（生命周期、持有内容、与 Trip 协作关系），但基础设施层设计自身未说明：(a) EdgeSessionContext 在 Spring Boot 边缘应用中的实例化方式（`@Scope("session")` 原型 Bean 还是编程式创建）；(b) 其生命周期与 Trip、VehicleStateBuffer、PhysiologicalDataBuffer 的具体绑定时机和管理机制（由哪个组件在点火信号到达时创建，熄火时由哪个组件销毁）；(c) 边缘侧进程重启后（如 OTA 刷写重启）EdgeSessionContext 的恢复策略。
- **改进建议**：在 §3.4 或新增独立小节中，从基础设施实现视角补充 EdgeSessionContext 的管理机制：创建触发（`MonitoringSessionStartedEvent` 或 `startMonitoringSession` 调用）、Spring 集成方式、与 VehicleStateBuffer/PhysiologicalDataBuffer 的实例绑定关系、销毁时机和资源清理逻辑。

### 问题 6：边缘侧仓储实现仅描述了 1/5，其余 4 个仓储的边缘实现缺失

- **所在位置**：§3.2.2
- **严重程度**：**中等（完整性不足）**
- **问题描述**：需求 §七.1 明确边缘侧需部署"边缘侧数据本地持久化（SQLite…存储 Trip 和告警）"；Decision 4 明确"边缘侧 SQLite 的 persistence 实现与云端 JPA 实现共用同一仓储接口"。但设计中仅描述了 `RoadRageVoiceRecordRepository.findByExpiryBefore` 的边缘实现（§3.2.2）。**TripRepository**（边缘侧 AlertPersistenceService 需通过它创建 SafetyAlertEvent）、**DriverRepository**（边缘侧需更新评分）、**VehicleRepository**（边缘侧需更新传感器状态和 OTA 状态）、**SystemAccountRepository**（边缘侧家属权限查询）的边缘侧 SQLite 实现策略均未描述。
- **改进建议**：在 §3.2 中增加"边缘侧仓储实现"小节，明确各仓储在 SQLite 环境下的实现策略——是采用 JDBC 直接操作 SQLite（如 Decision 4 所述），还是采用 JPA + Hibernate SQLite 方言。至少需说明边缘侧 TripRepository 的 CRUD 实现方式（SQLite 不支持 JPA `@ElementCollection` 和 `@Version` 的自动处理，需替代方案）。

### 问题 7：MQTT Topic `iot/family/...` 的用途与通信拓扑不一致

- **所在位置**：§3.5.1（MQTT Topic 路由设计表）
- **严重程度**：**轻微（歧义）**
- **问题描述**：Topic 路由表中定义了 `iot/family/{accountId}/alert` 和 `iot/family/{accountId}/status` 两条下行 Topic，方向标注为"下行"、QoS 分别为 1 和 0。但 §3.5.1 末尾的"家属 APP 连接方式"说明中明确"家属 APP 不直接连接 IoTDA MQTT——而是通过 WebSocket 长连接直连云端 Spring Boot 应用服务"。若家属 APP 不连接 MQTT，这两个 Topic 的消费方是谁？可能意图是 IoTDA 规则引擎将这些 Topic 的消息路由至云端应用服务（再由应用服务通过 WebSocket 推送给家属 APP），但设计中未说明此路由机制，与 MQTT 信道拓扑图（§3.7.2）也不匹配——该图仅显示家属 APP ↔ WebSocket ↔ 应用服务。
- **改进建议**：明确这两个 Topic 的用途——若为内部路由 Topic（IoTDA → 云端应用服务消费后转 WebSocket 推送），在 Topic 表中增加"消费方"列说明并补充路由机制；若确实不需要（家属推送完全走 WebSocket + Push Kit 通道），则删除这两个 Topic。

### 问题 8：部署架构图缺失

- **所在位置**：需求 §七.5 要求 vs 设计 §3.7
- **严重程度**：**轻微（格式缺失）**
- **问题描述**：需求 §七.5 明确要求提供"以 Mermaid flowchart 或文本图描述"的部署架构图。设计 §3.7.2 提供了 MQTT 信道拓扑图（ASCII 文本图），但该图仅覆盖 MQTT 通信层面，不是完整的部署架构图。§3.7.1 仅以表格形式列出组件清单，缺少一张描述所有组件之间网络连线关系（含协议标注）的整体架构图——如车载终端内部组件的进程内调用、车载终端 → IoTDA MQTT/TLS、云端应用服务 → 金仓数据库/TCP、Spring Boot → Redis/TCP 等。
- **改进建议**：增加一张 Mermaid flowchart 或 ASCII 文本图，覆盖 §3.7.1 列出的所有组件及其之间的网络连接关系（含传输协议标注）。

### 问题 9：VO-16 DrivingBehaviorCounters 映射归类不准确

- **所在位置**：§3.1.3（值对象嵌入映射策略）
- **严重程度**：**轻微（归类错误）**
- **问题描述**：VO-16 DrivingBehaviorCounters 被列在"策略 B：集合嵌入聚合根表（`@ElementCollection` + `@CollectionTable`）"分类下，但其映射方式实际描述为"嵌入列字段（`hard_braking_count` + `hard_acceleration_count` INTEGER）"——这是策略 A（`@Embedded` 嵌入列字段）的映射方式，而非策略 B 的 `@ElementCollection` 独立关联表方式。VO-16 不是集合类型值对象，是两个计数字段嵌入 Trip 表，应归于策略 A。
- **改进建议**：将 VO-16 移至策略 A 表格中，并说明其与 TripScore 同为 Trip 表的嵌入列字段。

### 问题 10：各表缺少显式主键生成策略说明

- **所在位置**：§3.1.1（聚合根表结构）
- **严重程度**：**轻微（实现细节缺失）**
- **问题描述**：需求 §一.1 要求说明各表的**主键策略**。设计中各表均标注了 `PK, NOT NULL` 和类型，但未说明主键的生成策略——是应用层生成 UUID 后赋值、JPA `@GeneratedValue` 自增序列、还是使用金仓的 `UUID` 生成函数。特别是 `trip_physiological_snapshot` 集合表使用"trip_id + timestamp 复合主键"、`guardianship` 表使用"复合主键 `(driver_id, account_id)`"、`alert_projection` 使用 BIGSERIAL——这些不同策略适用于不同场景，但正则聚合根表的主键策略未统一说明。
- **改进建议**：在各表定义中增加"主键策略"行或在 §3.1.1 开头统一说明：聚合根表主键采用应用层生成 UUID（通过 `@Id` + 构造器赋值），无 `@GeneratedValue`，理由：UUID 支持分布式无冲突生成，适合云边两侧独立创建实体后同步的场景。

### 问题 11（新增）：outbox 表缺少指数退避重试所需的 `last_attempt_at` 字段

- **所在位置**：§3.3.2（outbox 表结构） vs §3.3.5（重试策略）
- **严重程度**：**中等（实现阻塞）**
- **问题描述**：§3.3.5 描述了指数退避重试策略——"各事件的单次投递失败后，不立即重试同一事件，而是由下一轮轮询（已过退避时间）再次投递。退避间隔 = `min(2^retryCount × 1s, 60s)`"。但 outbox 表（§3.3.2）仅有 `created_at`（记录创建时间）和 `retry_count`，没有记录最近一次重试尝试时间的字段。OutboxRelayer 无法仅凭 `created_at` 和 `retry_count` 判断当前时刻是否已过退避等待期——它需要 `last_attempt_at` 或 `next_retry_at` 字段来精确控制各事件的重试间隔。当前设计下，OutboxRelayer 要么在每轮轮询中对所有未投递事件无差别重试（退避失效），要么需要额外内存状态跟踪各事件的最后尝试时间（增加复杂度且进程重启后丢失）。
- **改进建议**：在 outbox 表中增加 `last_attempt_at TIMESTAMP NULL` 字段（记录最近一次投递尝试时间），OutboxRelayer 查询条件改为 `WHERE published=FALSE AND (last_attempt_at IS NULL OR NOW() - last_attempt_at > retryDelay)`，其中 `retryDelay` 根据 `retry_count` 按指数退避公式计算。或改为 `next_retry_at TIMESTAMP NOT NULL` 字段，每次失败后更新为 `NOW() + retryDelay`，简化查询为 `WHERE published=FALSE AND next_retry_at <= NOW()`。

### 问题 12（新增）：需求 §一.4 要求乐观锁版本号在 JPA 中的实现方式和冲突处理策略全部说明，但 §3.1.4 未覆盖边缘侧 SQLite 的乐观锁实现

- **所在位置**：§3.1.4（乐观锁版本号统一规范）
- **严重程度**：**轻微（边缘实现缺失）**
- **问题描述**：§3.1.4 详细描述了 JPA 的 `@Version` 注解机制和 `OptimisticLockException` 转换策略，但该方案仅适用于云端 JPA（Hibernate）。边缘侧持久化使用 SQLite（Decision 4），而决策 4 同时指出边缘侧"不依赖 JPA（改用 JDBC 直接操作 SQLite）"。这意味着边缘侧无法使用 `@Version` 注解，需手动在 SQL 层实现 `UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?` + 检查受影响行数。§3.1.4 未说明边缘侧 SQLite 环境下乐观锁的等效实现方案。
- **改进建议**：在 §3.1.4 中补充边缘侧 SQLite 的乐观锁实现策略——在 JDBC 层手动构造带版本号条件的 UPDATE 语句，受影响行数为 0 时抛出 `OptimisticLockConflict`，保持与云端 JPA 相同的异常语义。

### 2.2 各核心模块质量评价
