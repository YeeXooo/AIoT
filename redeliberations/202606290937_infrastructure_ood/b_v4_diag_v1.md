## 问题清单

### 问题 1（严重）：IoTDA 上行消息消费与 outbox 写入之间存在事务边界断裂——消息丢失风险

- **所在位置**：§3.5.1「IoTDA 上行消息消费适配器」 + §3.3.1 云端事件总线实现
- **问题描述**：`IotdaUplinkConsumer` 从 DMS Kafka 消费 IoTDA 转发的上行消息后，调用 `EventBus.publish()` 将边缘侧领域事件写入 `domain_event_outbox` 表。这两个操作跨两个独立的事务资源——Kafka 消费（偏移量提交）和数据库（outbox 写入）。分属不同的本地事务，存在双重写入问题（dual-write problem）：
  - 场景 A：outbox 写入成功，Kafka 偏移量提交失败 → 消息重复消费，幂等去重可兜底（可接受）
  - 场景 B：outbox 写入失败（数据库异常），Kafka 偏移量已提交 → 消息**静默丢失**，边缘侧上报的告警事件永久丢失
- **严重程度**：严重——告警事件静默丢失直接影响安全告警的家属通知和看板聚合，且不可自愈
- **改进建议**：
  - (a) 明确 `IotdaUplinkConsumer` 的 Kafka 消费采用手动提交偏移量模式（`enable.auto.commit=false`），仅 outbox 写入成功后才提交偏移量；outbox 写入失败则抛出异常触发 Kafka 重平衡重试；
  - (b) 在 IotdaUplinkConsumer 内部维护一个本地重试队列——outbox 写入失败时暂存于内存/Redis 中，后台异步补偿重试写入，偏移量正常提交（接受最终一致性）；
  - (c) 补充分布式事务方案作为保障路径（如 Seata AT 模式协调 Kafka 消费事务与数据库事务）。

### 问题 2（严重）：语音存证授权解密依赖云端 DEW 下发临时密钥，但事故后车载终端可能离线导致解密链断裂

- **所在位置**：§3.6.2「授权解密」段落 + §3.5.5 DEW 密钥管理
- **问题描述**：设计规定授权解密需云端 DEW 下发临时解密密钥（"单次使用，用完即弃"），传输依赖 MQTT 下行通道。但语音存证的法定使用场景（交通事故定责、路怒投诉、司法协查）通常发生在车辆已停止运行甚至严重损坏之后——此时车载终端大概率处于离线状态（断电/损毁/信号盲区），无法接收 DEW 下发的临时密钥。设计未给出离线解密路径，存在**法律合规性风险**——存证加密保护了隐私但无法在合法场景下解密，与需求 §六.2 列出的三类授权场景的可用性要求矛盾。
- **严重程度**：严重——涉法涉规场景下的功能不可用，直接影响系统合规性
- **改进建议**：
  - (a) 增加**离线解密路径**：在云端保存每个 recordId 对应的加密会话密钥副本（以设备密钥包裹存储于 OBS），授权解密时由云端 DEW 解密设备密钥后再解开会话密钥，在云端完成解密后通过安全通道传输明文音频（全程 TLS + 仅内存暂存）；
  - (b) 增加**现场解密路径**：在边缘设备上设计本地紧急解密机制（如物理安全令牌 + DEW 预下发的离线授权码），供现场执法人员使用；
  - (c) 明确三种授权场景下的解密流程端到端路径，标注哪些依赖网络连通性、哪些有离线方案。

### 问题 3（一般）：OutboxRelayer 多实例部署下的重复投递缺乏数据库层并发排他机制

- **所在位置**：§6.3 并发策略表「Outbox 投递器多实例部署」行 + §4 场景 2 步骤 3
- **问题描述**：§6.3 称"每条 outbox 记录的投递由 `published=FALSE + last_attempt_at` 索引保证被唯一实例获取和处理。投递器在 UPDATE `published=TRUE` 时加 `WHERE published=FALSE` 条件，避免多实例重复投递"。此描述存在逻辑漏洞——SELECT 查询获取待投递事件列表后，至 UPDATE `published=TRUE` 之间存在时间窗口。两个 OutboxRelayer 实例可能同时查询到同一条记录、各自投递至 Kafka，然后各自的 UPDATE 仅一个成功（另一个 WHERE published=FALSE 匹配不上，受影响行数 0）。**但 Kafka 已经被投递了两次**。虽然消费方幂等去重可兜底（§3.3.5），但存在浪费带宽和增加下游处理冗余的问题。
- **严重程度**：一般——功能层面由消费方幂等去重兜底，但设计声称的"保证被唯一实例获取"不成立，可能误导实现者
- **改进建议**：
  - (a) 将 SELECT 查询改为 `SELECT ... FOR UPDATE SKIP LOCKED`，利用数据库行锁在轮询阶段即排他锁定待投递行；
  - (b) 或明确承认 at-least-once 语义包含投递层面的重复（非仅消息队列层的重试），将该限制写入设计文档而非声称"保证唯一实例获取"。

### 问题 4（一般）：fleet_dashboard_projection 事件驱动即时增量聚合缺少读-计算-写竞争保护

- **所在位置**：§3.2.3 P2「缓存/物化视图策略」路径 A
- **问题描述**：路径 A 采用事件驱动即时重算——当告警事件到达时，从 `alert_projection` 表中读取当前活跃告警集，计算 `driver_count` 和 `heatmap_data`，然后以 `INSERT ... ON CONFLICT DO UPDATE` 写入看板投影表。两个告警事件几乎同时到达（同一车队、同一风险等级/类型组合）时，两个线程可能读到相同的 `alert_projection` 快照、各自计算、然后串行写入——第二个写入覆盖第一个。虽然外层以 `ON CONFLICT DO UPDATE` 保证最后一次写入的原子性，但**写覆盖的是另一个线程基于已过期快照计算的结果**，导致看板数据可能不是严格最新的聚合（丢失了第一个线程已纳入但第二个线程未察觉的新告警）。设计应评估此读-计算-写窗口的语义后果——最坏情况是丢失一笔聚合更新，而定时全量刷新（路径 B，5 分钟）将其修正。如果业务上 P99 3s 的 SLO 可容忍偶发的 ≤5min 偏差，则应在设计文档中**显式声明这一取舍**。
- **严重程度**：一般——存在正确性风险但已有兜底机制，需明确声明而非隐含依赖
- **改进建议**：在 P2 的路径 A 描述中增加一句说明："注：并发告警事件可能导致读-计算-写竞争，最坏情况某次增量聚合结果被覆盖丢失，由路径 B 在下一个 5 分钟周期修正。P99 ≤3s SLO 接受此偶发偏差。"

### 问题 5（一般）：需求 §六.1 脱敏校验门控的具体实现位置和触发时机未明确——可编码性不足

- **所在位置**：§3.6.1「脱敏校验门控」段落
- **问题描述**：设计描述："在数据上云 MQTT 通道之前，基础设施层校验数据体中不包含 `raw_image` 字段（以 JSON Schema 验证）。若检测到原始图像数据，拒绝上云并记录安全审计日志"。此描述停留在意图层面——未说明门控的具体挂载点：(a) 是作为一个 MQTT Publisher 的发送前拦截器？(b) 还是作为独立的消息预处理管道组件？(c) JSON Schema 文件放在哪里？(d) 校验失败后的行为除审计日志外，是否通知判定引擎降级（缺失视觉通道）？(e) 这个门控逻辑归属 `infra.security` 还是 `infra.edge`？这些细节缺失使得开发者无法直接从设计文档开始编码。
- **严重程度**：一般——架构设计层概念清晰但实现指导不足
- **改进建议**：补充门控的组件归属（如 `SecurityGateFilter`，`@Component`，归属 `infra.security`）、挂载方式（MQTT Publisher 的 outgoing interceptor 或独立的 `DataSanitizer` 前置处理器）、Schema 文件路径（如 `classpath:schemas/sensor-data-schema.json`）、以及校验失败后的完整行为链。

### 问题 6（一般）：家属 APP 多通道消息推送缺乏去重设计——用户可能收到重复通知

- **所在位置**：§3.4.6 NotificationPort + §3.5.3 SMN 模板 + §3.7.2 部署图
- **问题描述**：家属端告警通知存在三条并行推送通道：(a) WebSocket 长连接实时推送（§3.7.2 描述家属 APP 通过 WebSocket 接收状态推送和告警通知）；(b) SMN 模板消息推送（§3.4.6，经华为云 SMN）；(c) Push Kit 系统级推送（§3.5.3，用于 APP 后台/未启动场景）。当家属 APP 处于**前台且 WebSocket 连接活跃时**，同一条告警如果同时触发了 WebSocket 推送 + SMN 推送（然后通过 SMN→Push Kit 再次送达），家属 APP 将收到重复通知。设计未描述去重策略或通道选择的优先级逻辑。
- **严重程度**：一般——影响用户体验但非功能错误
- **改进建议**：明确三通道的推送优先级策略：(a) 家属 APP 前台时仅走 WebSocket，不走 SMN/Push Kit；(b) 家属 APP 后台时走 Push Kit（含 SMN 载体）；(c) 家属 APP 侧基于 `event_id` 做本地去重（最近 N 条，类似云端 Redis LRU 策略）。在 NotificationPort 适配器内部维护家属 WebSocket 在线状态，按连接状态路由推送通道。

### 问题 7（一般）：OTA 升级固件签名的公钥来源和存储未说明

- **所在位置**：§3.4.5「车载终端侧 OTA 客户端（边缘侧）」组装与刷写段落
- **问题描述**：设计要求："刷写前校验固件包整体数字签名（RSA-2048），签名验证通过后触发系统重启"。但未说明验证签名的公钥从何而来、如何预置到车载终端、如何防范公钥被篡改或替换。若公钥以明文形式存储在文件系统中，攻击者可能通过替换 OTA 包和公钥文件绕过签名验证——这是一个安全设计缺口。
- **严重程度**：一般——安全机制存在但缺少可信根设计
- **改进建议**：补充公钥管理方案。建议路径：(a) 公钥在设备出厂时预烧录至 TEE/KeyStore 的安全存储区域（不可篡改）；(b) 或通过设备主密钥（DEW 下发）对公钥进行 HMAC 签名保护，刷写前先验公钥完整性再验固件签名；(c) 标注为本期模拟阶段暂不实现完整可信链，但给出占位设计。

### 问题 8（一般）：Guardianship 表"部分唯一索引"的金仓兼容性未经验证

- **所在位置**：§3.1.2 guardianship 关联表 + §6.3 并发策略
- **问题描述**：设计指定 `(driver_id, account_id)` 组合上建**部分唯一索引** `WHERE revoked_at IS NULL`，这是 PostgreSQL 9.x+ 的语法特性。但项目使用金仓数据库（Kingbase），虽然兼容 PostgreSQL 协议，但 "协议兼容" 不保证语法特性兼容——部分唯一索引（Partial Unique Index）是 PostgreSQL DDL 扩展语法，金仓的 PostgreSQL 兼容模式是否支持此语法未经验证。若金仓不支持，guardianship 的并发授予唯一性约束将失效，可能产生重复的活跃监护关系行。
- **严重程度**：一般——依赖可能不存在的数据库功能
- **改进建议**：
  - (a) 补充金仓版本号及已验证支持的部分唯一索引说明；
  - (b) 给出金仓不支持情况下的替代方案——如在应用层通过 `SELECT ... FOR UPDATE` 先锁定 `(driver_id)` 的所有活跃 guardianship 行再执行 INSERT，用悲观锁替代部分唯一索引。

### 问题 9（轻微）：需求 §六.2「二次校验」要求的具体操作步骤未充分展开

- **所在位置**：§3.6.2「清除」段落
- **问题描述**：需求 §六.2 明确："清除操作需经二次校验（验证文件确实过期，防止误删）"。设计的实现描述为："验证文件对应的 RoadRageVoiceRecord 元数据确认 `expiryTime < NOW()` 且 `isSealed = TRUE`"。这只解释了校验的**条件**（检查元数据），但未说明校验的**步骤**（是一步校验元数据即认为满足"二次"？还是元数据校验 + 文件系统时间戳交叉验证？）。"二次校验"的语义可被解释为仅一条 SQL 检查，使得防误删力度弱于需求预期——如果元数据被错误修改为已过期（如数据库 BUG 或运维误操作），单步校验无法阻止误删。
- **严重程度**：轻微——需求语义解释存在缩小化风险
- **改进建议**：明确"二次校验"的具体两步骤——(a) 一级校验：RoadRageVoiceRecord 元数据（`expiryTime < NOW() AND isSealed = TRUE`）；(b) 二级校验：文件系统层面验证加密文件的实际创建时间戳早于预期保留期限，且文件未被外部程序打开/锁定。两步均通过后方执行物理删除。此设计的"二次"含义与需求原文的"二次校验"一致。

### 问题 10（轻微）：VO-09 VehicleStateSnapshot 分类为"不持久化"与 EdgeSessionContext 崩溃恢复中的序列化持久化行为矛盾

- **所在位置**：§3.1.3 策略 D「不持久化的值对象」 vs §3.4.2a EdgeSessionContext destroy() + 重启恢复
- **问题描述**：§3.1.3 将 VO-09 VehicleStateSnapshot 归为"不持久化的值对象（瞬时/派生）"，适用策略 D。但 §3.4.2a 描述 EdgeSessionContext destroy() 时"将 VehicleStateBuffer 和 PhysiologicalDataBuffer 中尚存的数据序列化写入 SQLite 本地持久化（供紧急救援事后回取）"，重启恢复时"从 SQLite 恢复最近一次持久化的缓冲快照"。这表明 VehicleStateSnapshot 实际上是会被持久化到 SQLite 的——尽管不是进入云端金仓主表。策略 D 的"不持久化"分类至少需要在说明中补充"不入金仓主表，但在边缘侧崩溃恢复时序列化至 SQLite"的限定。
- **严重程度**：轻微——分类表述不够精确，可能导致开发者误解 VehicleStateSnapshot 的内存生命周期
- **改进建议**：在策略 D 表格中 VO-09 的说明改为："瞬时值对象，仅在 ring buffer 内存中存在，不入金仓主表；边缘侧进程崩溃前可选择性序列化至 SQLite 供紧急回取。"

### 问题 11（轻微）：DriverDeactivatedEvent 的事件消费者仅标注云端，但边缘侧应同步清理该驾驶员的数据

- **所在位置**：§3.3.6 事件消费者注册表 DriverDeactivatedEvent 行
- **问题描述**：当驾驶员被注销（DriverDeactivatedEvent），云端执行权限清理和隐私清理。但边缘侧设备（车载终端）上可能仍缓存有该驾驶员的 `driver`、`system_account` 只读数据（§3.2.2a 描述为云端驱动的只读缓存）。设计未说明 DriverDeactivatedEvent 如何传播至边缘侧以触发本地缓存清理——根据当前设计，边缘侧缓存将在 SQLite 中残留已注销驾驶员的账户和个人信息，违反数据最小化原则。
- **严重程度**：轻微——数据残留风险但影响范围有限（边缘侧数据量小、设备物理可控）
- **改进建议**：为 DriverDeactivatedEvent 增加"边缘侧"消费方（归属 `infra.edge`），消费者收到事件后清除 SQLite 中该 driver 的 `driver`、`system_account`、`driver_health_profile` 缓存行和关联的 guardianship 记录。同时在 DriverDeactivatedEvent 的 Topic 设计（§3.3.4）和消费者注册表（§3.3.6）中同步更新。

### 问题 12（轻微）：断网缓冲模块与 MQTT 客户端的集成接口未定义

- **所在位置**：§3.7.4「数据缓冲」+ §3.7.1 边缘侧组件表「断网缓冲模块」
- **问题描述**：断网缓冲模块被描述为独立的 Java 组件（"Java + 内存队列"），负责在 MQTT 断连时缓存待上报消息。但其与 MQTT 客户端的交互方式未定义——是作为 MQTT 客户端的 Decorator/Wrapper 拦截 `publish()` 调用？还是 MQTT 客户端检测到断连后主动将消息路由至缓冲模块？还是通过 MQTT 客户端的 `MqttCallback.connectionLost()` 回调切换数据通路？这些接口细节缺失使得开发者无法确定缓冲模块的具体实现方式。
- **严重程度**：轻微——可编码性受影响但可通过 MQTT 客户端的标准回调机制推断
- **改进建议**：增加一句说明缓冲模块与 MQTT 客户端的集成方式：推荐通过 Spring Integration MQTT 的 `MqttPahoMessageHandler` 包装一个 `OfflineBufferingDecorator`，在 `connectionLost` 回调时启用内存队列缓冲，在 `connectionComplete` 回调时按 FIFO 顺序排空队列。给出伪代码级别的接口契约。
