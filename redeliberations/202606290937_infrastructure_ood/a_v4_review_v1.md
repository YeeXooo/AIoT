# OOD 设计方案审查报告（v1）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 设计方案中所有类型形态选择均在 Java 语言类型系统能力范围内：JPA `@Entity` / `@Embeddable` / `@ElementCollection` / `@Version` 为标准 JPA 规范；Spring Data JPA 的 `JpaRepository` 继承、`@Query`、`@Modifying`、`EntityManager` 均为 Spring 生态标准能力；`ArrayDeque`、`ConcurrentHashMap`、`UUID` 为 JDK 标准库；ShardingSphere-JDBC 一致性哈希分片算法为官方内置能力。继承与实现关系（仓储实现类实现领域层接口、适配器实现端口接口）遵循 Java 单继承多接口实现约束，无违反类型系统限制的设计。

### 2. 标准库与生态覆盖

**[通过]** 设计依赖的能力均在目标栈覆盖范围内：金仓数据库通过 `PostgreSQLDialect` 兼容；HikariCP 为 Spring Boot 默认连接池；ShardingSphere-JDBC 嵌入模式为成熟方案；华为云 IoTDA / DMS Kafka / SMN / SparkRTC / DEW / OBS / DCS Redis 均有官方 SDK 或标准协议支持；BouncyCastle 依赖（HKDF-Expand）已在 §3.6.2 中标明。设计中对各库能力的假设（如 ShardingSphere-JDBC 的绑定表、广播表、HintManager 功能）合理，属于其文档声明的能力范围。

**[轻微]** §3.5.2 广播表本地事务原子性的待验证项标注了 ShardingSphere-JDBC 目标版本为 5.x，版本号较模糊（主版本范围），建议在编码阶段明确锁定具体 5.x 小版本号以便可复现验证。

### 3. 语言特性可行性

**[通过]** 错误处理策略与 Java unchecked exception 模式匹配——仓储层抛出 `PersistenceException` 及其子类型，适配器层按场景抛出对应业务异常，outbox 投递器的内部失败由状态机管理不向上传播。并发设计合理：边缘侧单线程串行（确定性实时响应），云端侧乐观锁 + Kafka 分区 + Redis 幂等去重的组合与 Java 并发模型兼容。资源管理方案（HikariCP 连接池、EdgeSessionContext 生命周期管理、MQTT 重连）均在 Java/Spring 资源管理模式内可行。模块结构遵循 Maven/Gradle 标准多模块项目组织方式，依赖方向单向。

### 4. 设计一致性

**[通过]** 各抽象职责描述清晰无歧义——§二模块划分表明确了 8 个模块的职责、领域契约依赖和部署位置。协作关系形成闭环：从聚合根持久化 → outbox 事件写入 → Kafka 投递 → 消费者处理 → 投影写入 → 缓存失效，全链路在 §3.2.3、§3.3、§4 场景 2/3 中有完整描述。行为契约在 §四中以 4 个端到端场景覆盖了关键交互路径。模块间依赖方向合理（仅依赖领域层接口契约，不依赖应用层），无循环依赖。

**[通过]** v4 迭代需求中的 7 个问题已全部解决并逐条验证：
- 问题 1（outbox 广播表原子性）：§3.5.2 已新增待验证项说明，含三种路径；
- 问题 2（跨分片 JOIN）：§3.2.3 P2 已改为基于 `alert_projection`（shard-0）聚合；
- 问题 3（投影表写路由）：§3.5.2 已新增 HintManager 推荐方案 + 独立 DataSource 备选方案；
- 问题 4（标题版本号）：文档标题已修正为 v4；
- 问题 5（@ElementCollection 冻结风险）：§3.1.3 已改为 StatelessSession 推荐方案，标注技术风险；
- 问题 6（ArrayDeque 固定容量）：§3.4.1/§3.4.2 已补充容量检查 + removeFirst() 策略；
- 问题 7（destroy 时序安全）：§3.4.2a 已新增安全观察窗口（默认 5s）机制。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则——§二将基础设施层拆分为 8 个职责明确的模块，各模块仅实现领域层声明的接口契约。抽象层次恰当：设计聚焦于架构级决策（分片策略、事件总线方案、端口适配器选型），将实现细节（具体方法签名、字段级映射）留给后续详细设计，符合架构级 OOD 的抽象层次。设计便于测试：所有组件基于接口契约编程，仓储、端口、事件总线均可通过注入 mock 实现进行隔离测试；乐观锁冲突、MQTT 断网、outbox 投递失败等异常路径在 §五中明确了错误表达方式，便于异常路径测试用例设计。

**[轻微]** §3.1.5 分片数量初始值（4 片）的选型依据未给出（如预期数据量、单分片容量上限），建议补充简要的容量预估说明，使后续扩容决策有据可依。

## 修改要求

无。审查结论为 APPROVED，不存在严重或一般问题。
