## 问题清单

### 问题 1：`domain_event_outbox.event_id` 字段类型与统一主键策略不一致

- **所在位置**：§3.3.2（第 641 行）vs 第 7 行（统一主键策略）
- **严重程度**：一般
- **问题描述**：文档第 7 行声明的统一主键策略规定"所有聚合根表……及独立实体表……的主键均采用应用层生成 UUID（`java.util.UUID.randomUUID().toString()`），存储为 `VARCHAR(64)`"。但 `domain_event_outbox` 表的 `event_id` 字段类型为 `UUID`（PostgreSQL 原生类型），与统一策略不一致。该表虽非聚合根表，但其主键 `event_id` 同样由应用层生成 UUID 值，应遵循统一存储约定。相比之下，`domain_event_dlq.dlq_id` 已在 v11 中修正为 `VARCHAR(64)`，outbox 表应当同步修正。此外，金仓数据库对 PostgreSQL `UUID` 原生类型的兼容性本身也是待验证项（PostgreSQL 10 引入的 `UUID` 类型金仓是否完全支持）。
- **改进建议**：将 `domain_event_outbox.event_id` 类型改为 `VARCHAR(64)`，与 `domain_event_dlq.dlq_id` 和统一主键策略保持一致；或在统一主键策略中补充例外说明（outbox 表因无须与其他表 JOIN 而允许使用原生 UUID 类型）。

### 问题 2：分片环境下的外键约束可行性未讨论

- **所在位置**：§3.1.6 DDL（第 411-431 行、第 437-454 行）vs §3.1.5 分片策略（第 399-403 行）
- **严重程度**：一般
- **问题描述**：§3.1.6 概念级 DDL 中 Trip 表声明了 `FOREIGN KEY (driver_id) REFERENCES driver(driver_id)` 和 `FOREIGN KEY (vehicle_id) REFERENCES vehicle(vehicle_id)`；SafetyAlertEvent 表同样声明了 `FOREIGN KEY (trip_id) REFERENCES trip(trip_id)` 和 `FOREIGN KEY (driver_id) REFERENCES driver(driver_id)`。然而根据 §3.1.5，`driver` 和 `vehicle` 为广播表（每个分片冗余一份），`trip` 为按 `vehicle_id` 分片的分片表。在 ShardingSphere-JDBC 管理多物理数据库的部署中，由于被引用表（如 `driver`）的副本分布在多个物理数据库上，跨分片的物理外键约束是否能在数据库层实际创建并生效未经说明。若 ShardingSphere-JDBC 不支持在分片表上创建指向广播表的物理外键，上述 DDL 中的 FOREIGN KEY 约束可能仅具文档意义而无法实际执行，编码实现时产生困惑。
- **改进建议**：在 §3.1.6 或 §3.1.5 中补充说明：FK 约束在分片环境下属于逻辑约束（文档级声明），实际数据库层面不创建物理外键（由应用层/ShardingSphere-JDBC 保证引用完整性）；或确认 ShardingSphere-JDBC 目标版本 5.x 支持在分片表上创建指向广播表的物理外键并注明验证状态。

### 问题 3：guardianship 并发控制方案中 Redis 锁为单点故障，缺少降级路径

- **所在位置**：§3.1.2（第 274 行）
- **严重程度**：一般
- **问题描述**：§3.1.2 针对 guardianship 广播表的并发权限变更推荐使用 Redis `SET NX` 分布式锁（锁键 `(driver_id, account_id)`，锁超时 10s），理由是广播表上的 `SELECT ... FOR UPDATE` 不具备跨分片排他性。然而该方案引入 Redis 作为 guardianship 权限写操作的关键路径依赖——若 Redis 服务不可用（网络故障、Redis 集群宕机），所有 guardianship 操作（权限授予/撤销）将因无法获取锁而全部失败。这在紧急救援场景下（如家属需要紧急获取监护权限以发起远程对讲）可能产生安全后果。设计中未给出 Redis 不可用时的降级路径。
- **改进建议**：补充 Redis 不可用时的降级策略：（a）短期 Redis 故障（≤30s）由调用方重试；（b）Redis 长期不可用时降级为"无锁模式"——依赖 guardianship 表的部分唯一索引（若金仓支持）或 `INSERT ... ON CONFLICT` 的数据库级约束兜底，接受最终一致性窗口期内可能出现短暂重复活跃行，由后续定期清理任务修正。同时将降级行为通过 SMN 告警通知运维。

### 问题 4：数据脱敏校验仅在 MQTT 上行路径执行，未覆盖本地持久化路径

- **所在位置**：§3.6.1（第 1042-1048 行）vs §3.7.4（边缘侧本地持久化）
- **严重程度**：一般
- **问题描述**：§3.6.1 脱敏校验门控（`SecurityGateFilter`）的挂载点为 MQTT 上行消息发布链路的前置拦截点，校验"数据上云 MQTT 通道之前"的消息体中不包含 `raw_image` 字段。但需求 §六.1 要求"原始图像帧在关键点提取完成后立即丢弃，不写入任何持久化存储"——此要求包含两个维度：(a) 不上云（MQTT 上行通道，已覆盖）；(b) 不写入本地持久化存储（边缘侧 SQLite，未覆盖）。当前设计中，脱敏校验仅在 MQTT 上行路径执行。若某条数据通路将特征数据写入 SQLite（如断网缓冲场景下 Trip 状态数据本地落盘），未经脱敏校验的原始图像可能通过本地持久化路径意外留存，违反需求中的"不写入任何持久化存储"约束。
- **改进建议**：将脱敏校验门控扩展至数据写入 SQLite 之前的持久化路径上，或在 §3.6.1 中补充声明：本地持久化写入同样经过 `SecurityGateFilter` 校验（可挂载于 JDBC 写入拦截点或数据采集→持久化的数据通路），确保原始图像字段不会出现在任何持久化存储中。

### 问题 5：Vehicle 表 `fleet_id` 字段的写入时机和管理组件未定义

- **所在位置**：§3.1.1 Vehicle 表（第 143 行）vs §3.7.1 组件部署表
- **严重程度**：轻微
- **问题描述**：Vehicle 表定义中 `fleet_id VARCHAR(64) NOT NULL` 字段的说明仅为"车辆所属车队的标识，供看板投影按车队聚合查询"。但该字段由谁在何时写入（车辆注册时由 `infra.cloud` 模块写入？还是由车队管理员通过应用层操作写入？还是从 IoTDA 设备元数据中获取？），以及写入之后是否可变（车辆是否可能更换所属车队），均未明确。而 `fleet_dashboard_projection` 投影表（§3.2.3 P2）和 `alert_projection` 的冗余 `fleet_id` 字段均依赖 Vehicle 表中的 `fleet_id` 值，若该字段为空或错误将导致看板聚合结果不正确。
- **改进建议**：在 Vehicle 表定义的上下文（§3.1.1 或 §3.5.1）中补充 `fleet_id` 的来源（车辆注册初始化时由 `infra.cloud` 模块从 IoTDA 设备元数据或车队管理配置中获取写入）、可变性（不可变或需经车队管理员审批后变更）和变更时对下游投影表的级联影响声明。

### 问题 6：权限授予的"撤销后再授予"场景下 `guardianship` 新旧行的时序语义未定义

- **所在位置**：§3.1.2（第 266-268 行）vs §3.1.4 不适用场景说明（第 381 行）
- **严重程度**：轻微
- **问题描述**：guardianship 表以 `(driver_id, account_id, granted_at)` 复合主键保留历史记录，每次授予创建新行，撤销以 `UPDATE revoked_at = NOW()` 更新当前活跃行。但场景"家属权限被撤销后又重新授予"（同一 driver + account 在撤销后再授权）时，旧行 `revoked_at` 已被更新为非 NULL（表示已撤销），新行以新的 `granted_at` 创建且 `revoked_at = NULL`。此时存在两条记录——一条已撤销、一条活跃——语义上正确。但当"撤销→重新授予"发生在极短时间窗口内（如家属端错误撤销后立即重新授权），`granted_at` 的时间戳顺序是唯一的时序保证。设计中未声明应用层如何处理 `granted_at` 的生成（使用当前时间还是应用层显式传入），以及当两条记录的 `granted_at` 因时钟精度问题相同时的优先级判定规则，可能导致审计查询时对"哪次是当前有效授权"的判定歧义。
- **改进建议**：在 §3.1.2 guardianship 表定义中补充 `granted_at` 的生成约定（建议由应用层在创建行时显式赋值 `Instant.now()`，保证单调递增），并声明"当前有效授权"的判定规则为 `revoked_at IS NULL ORDER BY granted_at DESC LIMIT 1`。

### 问题 7：`MediaSessionPort` 的 `SessionType` 参数扩展超出需求范围但未记录变更理由

- **所在位置**：§3.4.8（第 910 行）vs 需求 §四.8
- **严重程度**：轻微
- **问题描述**：需求 §四.8 定义的 MediaSessionPort 端口方法契约为 `createSession(driverId, familyAccountId, sessionType)` 和 `endSession(sessionHandle)`——仅两个方法。设计 §3.4.5 将 MediaSessionPort 适配器的方法扩展为包含 `terminateSession`（与需求中的 `endSession` 语义等价但命名不同）和新增的 Token 续期机制。此外，需求中的 `sessionType` 参数未定义具体枚举值，设计中在 §3.4.8 引入了 `SessionType.VIDEO` 作为视频升级请求参数。方法命名的差异（`endSession` vs `terminateSession`）和新增的参数语义未在修订说明中记录，下游消费者在阅读设计文档时可能对端口契约的权威版本产生困惑。
- **改进建议**：统一端口方法命名与需求一致（或将 `terminateSession` 标注为 `endSession` 的适配器层内部实现名），在 §3.4.8 的方法列表中明确标注各方法对应于需求中哪个端口方法契约。`SessionType` 枚举值的扩展（如 `AUDIO_ONLY`、`VIDEO`）建议在端口方法契约概要中显式列出。

## 整体评价

本产出（v11）经过 5 轮迭代修复，在需求覆盖度、深度和完整性方面已达到较高水平。需求的七个章节均有对应的设计章节覆盖，接口定义基本满足下游编码实现需要。历史迭代中积累的严重问题（如 JPQL 矛盾、分片键缺失、outbox 事务断裂、离线解密断链等）均已得到有效修复。

当前识别的问题集中在以下三个维度：
1. **内部一致性问题**（问题 1）：outbox 表主键类型与统一策略不一致，为跨表统一处理主键带来困惑
2. **边界条件与异常路径覆盖不足**（问题 2、3、4）：分片环境下的外键可行性、Redis 锁降级、脱敏校验的本地持久化覆盖均存在缺口
3. **接口与数据的运维约定缺失**（问题 5、6、7）：`fleet_id` 写入时机、权限变更时序语义、端口方法命名对齐等细节影响可编码性

上述问题均不构成事实错误或逻辑矛盾，无阻塞级严重缺陷。建议在进入编码阶段前修正问题 1-4，问题 5-7 可在编码期间补充约定。
