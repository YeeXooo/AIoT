## 一、需求响应充分度

### 1.1 缺失需求 §一.5 要求的概念级 DDL 概要

| 属性 | 内容 |
|------|------|
| **问题描述** | 需求 §一.5 明确要求产出"金仓数据库的 CREATE TABLE DDL 概要（概念级，不含完整 DDL，但需展示关键表的结构定义）"。当前文档 §3.1.1-§3.1.2 仅以 Markdown 表格列出字段名和类型，未提供任何 DDL 语法级别的结构展示（如 `CREATE TABLE trip (...)` 形式的概要）。表格描述了列定义，但缺少表级约束（如 FOREIGN KEY 子句、CHECK 约束语法、索引的 CREATE INDEX 形式）的 SQL 表达，编码实现者无法直接从设计文档还原数据库 schema。 |
| **所在位置** | 需求 §一.5 vs 设计 §3.1.1-§3.1.3 |
| **严重程度** | 一般 |
| **改进建议** | 在 §3.1 末尾新增"概念级 DDL 概要"小节，以 `CREATE TABLE` 语法展示 1-2 张代表性表（如 `trip` 和 `safety_alert_event`），含完整列定义、PRIMARY KEY、FOREIGN KEY、UNIQUE、NOT NULL 等约束，以及代表性索引的 `CREATE INDEX` 语句。其余表可概述关键约束差异。 |

### 1.2 DriverHealthProfile 表设计缺少独立查询路径的显式声明

| 属性 | 内容 |
|------|------|
| **问题描述** | 需求 §一.3 明确指出 DriverHealthProfile（E-03）"不是聚合根但需要独立查询"。当前设计 §3.1.2 将其表主键设为 `driver_id`，通过 JPA `@OneToOne` + `@MapsId` 共享 Driver 主键。虽然该映射不阻止通过 JPQL 或原生 SQL 独立查询该表（`@OneToOne` 仅为 JPA 关系映射约定，`EntityManager.createQuery("FROM DriverHealthProfile WHERE bloodType = :t")` 仍然可执行），但设计中未显式声明独立查询的实现路径——即哪些查询场景需要绕过 Driver 聚合直接查询 `driver_health_profile` 表、通过 JPQL/原生 SQL 还是投影表实现。缺少此声明会使实现者因 `@OneToOne` + `@MapsId` 的强关联语义而默认通过 Driver 聚合加载，遗漏独立查询需求。 |
| **所在位置** | 需求 §一.3 vs 设计 §3.1.2（DriverHealthProfile 表结构） |
| **严重程度** | 一般 |
| **改进建议** | 在 §3.1.2 DriverHealthProfile 表描述中增加独立查询路径声明：明确哪些健康指标字段可被独立检索（如 `blood_type`、`chronic_history`），说明独立查询通过 `EntityManager.createNativeQuery` 直接执行而非经过 JPA 关系映射加载。若独立查询由 CQRS 投影表覆盖，则说明投影表与源表的同步策略。 |

## 二、事实错误与逻辑矛盾

### 2.1 SystemAccountRepository.findByDriver 的 JPQL 在 §3.2.1 与 §3.2.2 中矛盾

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.2.1 的仓储实现总览表中，`SystemAccountRepository.findByDriver` 的 JPQL 写为 `JOIN a.guardianships g`——这隐含 `SystemAccountEntity` 上存在一个 JPA `@OneToMany` 映射字段 `guardianships`。但 §3.1.2 明确 `guardianship` 是独立的关联表（多对多关系），不属于 SystemAccount 聚合的内部字段。而 §3.2.2 的正确 JPQL 为 `JOIN guardianship g ON a.id = g.accountId`（跨表 JOIN）。两条 JPQL 对同一查询给出了互相矛盾的实现方式，其中 §3.2.1 的写法无法直接编译通过（除非编码者在 Entity 上额外添加 `guardianships` 字段，但这与设计意图矛盾）。 |
| **所在位置** | §3.2.1 第 408 行 vs §3.2.2 第 425 行 |

## 三、深度与完整性

### 3.1 OTA 升级刷写期间安全监测中断的安全风险未被承认

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.4.5 描述 OTA 刷写阶段"车载终端暂停所有感知和判定业务（安全降级为仅保留 HMI 基本反馈），预计中断窗口 ≤2 分钟"。这意味着在固件刷写期间，碰撞失能检测、疲劳判定、路怒检测等全部安全功能停止运行，但设计未对"中断期间发生安全事件如何处理"给出任何陈述。这是一个系统性安全风险，需求文档和领域层均未授权 OTA 过程中可停机安全监测。作为基础设施层设计，至少应标注此安全风险并说明对应用层/领域层的约束要求（如：禁止在行驶中执行 OTA，或 OTA 仅允许在车辆驻车熄火状态下触发）。 |
| **所在位置** | §3.4.5 第 765 行 |
| **严重程度** | 一般 |
| **改进建议** | 补充 OTA 刷写前置条件声明：OTADeliveryPort 或 DS-15 应在触发刷写前校验车辆状态为驻车熄火（可通过 VehicleStateSnapshot 的 `doorLockStatus` 和 `speed` 判断），行驶中仅允许接收和暂存升级包、禁止进入刷写阶段。或明确标注此为已由需求方接受的已知风险并给出接受理由。 |

## 五、异常场景与边界条件覆盖

> 本节为响应任务要求"异常场景和边界条件是否已考虑"的专项审查。逐子系统检查主要异常路径是否已定义处理策略。

### 5.1 Outbox 表 / DLQ 表在消息队列长时间不可用时的存储膨胀

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.3.5 定义了 outbox 投递失败的重试策略（最多 10 次 → 移入 DLQ），但未讨论当 DMS Kafka 长时间不可用（如数小时级故障）时 outbox 表的写入速率和存储膨胀问题。若多车辆持续产生告警事件（每秒数十至数百条 outbox 写入），数小时内 outbox 表可能膨胀至千万行级别，影响数据库性能和轮询效率。DLQ 表同样面临无限增长风险——无 DLQ 自动清理策略（如按时间保留窗口清理旧死信记录）。 |
| **所在位置** | §3.3.2 outbox 表结构 + §3.3.5 重试策略 |
| **严重程度** | 一般 |
| **改进建议** | (a) 补充 outbox 表存储监控指标和告警阈值；(b) 当 `published = FALSE` 的积压行数超过阈值时触发降级——暂停非关键事件（如 TripScoredEvent、PerformanceWarningEvent）的 outbox 写入，仅保留安全关键事件；(c) 为 DLQ 表增加基于 `moved_at` 的时间保留窗口清理策略（如 30 天后自动归档或删除）。 |

### 5.2 边缘侧 SQLite 核心表无容量管理与清理策略

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.7.4 为断网缓冲的 MQTT 消息定义了"7 天保留窗口"的清理策略（`OfflineBufferCleanupTask`），但边缘侧 SQLite 的核心持久化表——`trip`、`safety_alert_event`、`road_rage_voice_record`——在数据成功同步至云端后，边缘侧的本地副本是否清理？若不清理，边缘侧运行数月后 SQLite 可能积累大量已完成的行程和告警记录，导致存储空间不足。设计 §3.7.4 末尾的"磁盘空间不足降级"仅描述删除"已成功上报并超过保留期的 SQLite 本地备份数据"，但未明确核心表的保留策略（保留最近 N 条行程还是按时间窗口清理）。 |
| **所在位置** | §3.7.4 存储空间监控段 vs SQLite 核心表 |
| **严重程度** | 一般 |
| **改进建议** | 补充边缘侧 SQLite 核心表的清理策略：已同步至云端且超过保留期的 Trip/SafetyAlertEvent 记录按 `created_at` 清理（如保留最近 30 天），RoadRageVoiceRecord 按 `expiry_time` 清理（已在 §3.6.2 定义）。明确清理触发方式（定时任务，如每日凌晨）。 |

### 5.3 IoTDA 离线消息缓存 24 小时上限与长期离线车辆的矛盾

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.7.5 和 §3.3.5 描述 IoTDA 离线消息缓存最长 24 小时——车辆离线超过 24 小时后，仍缓存在 IoTDA 的上行消息将丢失。但设计未定义：(a) 车辆离线超过 24 小时后重新上线时，云端如何获知丢失了哪些时段的数据？(b) 边缘侧在断网超过 24 小时后的行为——边缘侧本地缓冲是否也有上限？若边缘侧缓冲无容量上限但 IoTDA 已丢弃超过 24 小时的消息，将造成边缘侧认为已上报成功但云端实际未收到的假象。 |
| **所在位置** | §3.7.5（OTA 升级行 + IoTDA 缓存说明）、§3.7.4（边缘侧缓冲） |
| **严重程度** | 轻微 |
| **改进建议** | 明确边缘侧断网超过 24 小时场景的同步策略：由边缘侧本地缓冲（SQLite）兜底 IoTDA 的 24 小时缓存限制——断网恢复后，边缘侧不仅重发 MQTT 积压队列中的消息，还应扫描 SQLite 中 `queued_at` 超过 24 小时但未收到 `sync_ack` 的记录，通过批量上报补推。 |

### 5.4 DEW 密钥服务不可用时的边缘侧加密降级策略

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.5.5 描述 DEW 灾备时"边缘侧使用本地安全区域缓存的当前设备主密钥继续执行加密操作，解密操作排队等待 DEW 服务恢复"。这是一个合理的短期降级策略，但未定义：(a) 缓存的设备主密钥的有效期——若 DEW 服务长时间不可用（如数小时），缓存密钥是否仍然有效？(b) 若缓存密钥同时丢失（如边缘侧进程重启后 KeyStore 中密钥被清空但 DEW 不可用时无法重新获取），新产生的路怒语音存证如何加密？是否有"明文暂存 + 事后加密"的降级路径？ |
| **所在位置** | §3.5.5 灾备段 |
| **严重程度** | 轻微 |
| **改进建议** | 补充 DEW 不可用时的分级降级策略：(a) 短期不可用——使用缓存密钥加密（风险可接受）；(b) 长期不可用（如超过 1 小时）——触发安全审计告警，新存证以临时随机密钥加密存储，密钥以设备主密钥包裹后本地缓存，待 DEW 恢复后上传包裹密钥至云端；标注 DEW 不可用期间存证的合规风险（加密强度可能下降）。 |
