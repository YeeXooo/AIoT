## 当前审查结果

**本轮诊断报告来源**：`b_v4_diag_v2.md`（第 4 次迭代 · 第 2 轮审查），质询结论 **LOCATED**（诊断结论被确认）。

本诊断报告共发现 15 个问题，按严重程度分级如下：

### 严重问题（2 个，需优先修正）

1. **IoTDA 上行消息消费与 outbox 写入之间存在事务边界断裂——消息丢失风险**（§3.5.1 + §3.3.1）
   - `IotdaUplinkConsumer` 从 DMS Kafka 消费消息后写入 outbox 表，两者分属独立事务资源（Kafka 偏移量 vs 数据库），存在双重写入问题。场景 B（outbox 写入失败 + 偏移量已提交）导致告警事件静默丢失，不可自愈。
   - 改进建议：(a) 采用手动提交偏移量模式 `enable.auto.commit=false`，仅 outbox 写入成功后才提交偏移量；(b) 维护本地重试队列，outbox 写入失败时暂存内存/Redis 异步补偿重试；(c) 补充分布式事务保障路径（Seata AT 模式）。

2. **语音存证授权解密依赖云端 DEW 下发临时密钥，事故后车载终端可能离线导致解密链断裂**（§3.6.2 + §3.5.5）
   - 设计规定解密需 DEW 下发临时密钥经 MQTT 下行通道传至终端，但事故场景下终端大概率离线（断电/损毁），无法接收密钥。存在法律合规性风险。
   - 改进建议：(a) 增加离线解密路径——云端保存加密会话密钥副本（设备密钥包裹存储于 OBS），在云端完成解密后安全传输；(b) 增加现场解密路径——物理安全令牌 + DEW 预下发离线授权码；(c) 明确三种授权场景下的端到端解密路径。

### 一般问题（7 个）

3. **OutboxRelayer 多实例部署下的重复投递缺乏数据库层并发排他机制**（§6.3 + §4 场景 2 步骤 3）
   - §6.3 声称"保证被唯一实例获取"但实际 SELECT 到 UPDATE 之间存在时间窗口，两实例可能同时投递同一记录至 Kafka。
   - 改进建议：(a) 将 SELECT 改为 `SELECT ... FOR UPDATE SKIP LOCKED` 利用行锁排他；(b) 或明确承认 at-least-once 语义包含投递层面重复，修正设计文档中"保证唯一实例获取"的表述。

4. **fleet_dashboard_projection 事件驱动即时增量聚合缺少读-计算-写竞争保护**（§3.2.3 P2 路径 A）
   - 两个告警事件并发到达时，路径 A 存在读-计算-写竞争窗口，后写入可能覆盖前一次基于已过期快照的聚合结果。
   - 改进建议：在路径 A 描述中显式声明并发竞争取舍——"注：并发告警事件可能导致读-计算-写竞争，最坏情况某次增量聚合结果被覆盖丢失，由路径 B 在下一个 5 分钟周期修正。P99 ≤3s SLO 接受此偶发偏差。"

5. **脱敏校验门控的具体实现位置和触发时机未明确——可编码性不足**（§3.6.1）
   - 门控描述停留在意图层面，未说明挂载点（MQTT Publisher interceptor 还是独立前置处理器）、组件归属（`infra.security` 还是 `infra.edge`）、Schema 文件路径、校验失败后的完整行为链。
   - 改进建议：补充组件归属（如 `SecurityGateFilter`，`@Component`，归属 `infra.security`）、挂载方式、Schema 路径（如 `classpath:schemas/sensor-data-schema.json`）、校验失败后行为链。

6. **家属 APP 多通道消息推送缺乏去重设计——用户可能收到重复通知**（§3.4.6 + §3.5.3 + §3.7.2）
   - 家属端存在 WebSocket / SMN / Push Kit 三条并行推送通道，APP 前台时可能收到重复通知。
   - 改进建议：明确三通道优先级策略——前台仅走 WebSocket，后台走 Push Kit；APP 侧基于 `event_id` 本地去重（最近 N 条）；NotificationPort 适配器内部按连接状态路由推送通道。

7. **OTA 升级固件签名的公钥来源和存储未说明——缺乏可信根设计**（§3.4.5 车载终端侧 OTA 客户端）
   - 要求 RSA-2048 签名验证但未说明公钥来源和防护。
   - 改进建议：公钥预烧录至 TEE/KeyStore 安全区域，或通过设备主密钥 HMAC 签名保护。

8. **Guardianship 表"部分唯一索引"的金仓兼容性未经验证**（§3.1.2 + §6.3）
   - 部分唯一索引 `WHERE revoked_at IS NULL` 是 PostgreSQL DDL 扩展语法，金仓兼容性未验证。
   - 改进建议：补充金仓版本号及验证说明；给出金仓不支持时的悲观锁替代方案（`SELECT ... FOR UPDATE` 锁定后再 INSERT）。

9. **`trip_physiological_snapshot` 表列结构未显式定义——三路数据来源共享同一张表但字段无从获取**（§3.1.3 策略 B + §3.2.2a + §3.7.4）
   - 该表被 JPA `@ElementCollection`、溢出优化 JDBC batch INSERT、边缘侧同步 JDBC batch INSERT 三路独立数据路径使用，但完整列清单从未显式定义。
   - 改进建议：以表格形式定义该表的完整列结构（`trip_id`、`timestamp`、`heart_rate`、`blood_oxygen`、`emotion_index` 等），标注三路数据来源对各自写入列的义务。

### 轻微问题（6 个）

10. **需求 §六.2「二次校验」要求的具体操作步骤未充分展开**（§3.6.2「清除」段落）
    - 设计仅描述校验条件（元数据查过期）但未说明"二次"的两步骤含义。
    - 改进建议：明确两级校验：(a) 元数据校验（`expiryTime < NOW() AND isSealed = TRUE`）；(b) 文件系统层面验证（加密文件实际创建时间戳 + 文件未被外部锁定）。

11. **VO-09 VehicleStateSnapshot 分类为"不持久化"与 EdgeSessionContext 崩溃恢复中的序列化持久化行为矛盾**（§3.1.3 策略 D vs §3.4.2a）
    - 策略 D 标注 VO-09 为"不持久化"，但 destroy() 和崩溃恢复中会序列化至 SQLite。
    - 改进建议：策略 D 表格中 VO-09 说明改为："瞬时值对象，仅在 ring buffer 内存中存在，不入金仓主表；边缘侧进程崩溃前可选择性序列化至 SQLite 供紧急回取。"

12. **DriverDeactivatedEvent 的事件消费者仅标注云端，但边缘侧应同步清理该驾驶员的数据**（§3.3.6）
    - 驾驶员注销后边缘侧 SQLite 中残留 `driver`、`system_account` 等缓存数据。
    - 改进建议：为 DriverDeactivatedEvent 增加边缘侧消费方（归属 `infra.edge`），清除 SQLite 中该 driver 的缓存行。

13. **断网缓冲模块与 MQTT 客户端的集成接口未定义**（§3.7.4 + §3.7.1）
    - 缓冲模块与 MQTT 客户端交互方式（Decorator/Wrapper？回调切换？）未明确。
    - 改进建议：推荐通过 Spring Integration MQTT 的 `MqttPahoMessageHandler` 包装 `OfflineBufferingDecorator`，在 `connectionLost`/`connectionComplete` 回调时切换数据通路。

14. **边缘侧缓冲数据同步成功后缺乏清理触发机制——SQLite 本地存储可能无限增长**（§3.7.4）
    - "已成功上报"的判定依赖云端确认但缺少端到端确认机制。
    - 改进建议：(a) 补充端到端同步确认机制——云端回复 `sync_ack` 消息，边缘侧据此清理；(b) 或采用基于时间的保留窗口策略（SQLite 数据保留 7 天后物理删除）。

15. **§3.1.3 策略 B 集合冻结实现描述与修订记录 v8 声称的实现方式不一致——正文与修订记录矛盾**（§3.1.3 正文 vs §修订说明 v8）
    - 正文描述冻结为"不可变包装 + StatelessSession"，修订说明 v8 声称"置 null + detach() + merge()"，两处是不同策略。
    - 改进建议：将修订说明 v8 中的描述修正为与正文一致，或将正文更新为与修订说明一致的方案，二者必须统一。
