## 判定理由

组件B诊断报告（b_v4_diag_v2.md）共识别 15 个问题：2 个严重（#1 IoTDA 上行消息消费与 outbox 写入事务边界断裂、#2 语音存证离线解密链断裂）、7 个一般（#3 Outbox 并发排他缺失、#4 fleet_dashboard 读-计算-写竞争、#5 脱敏校验门控可编码性不足、#6 家属 APP 多通道推送去重缺失、#7 OTA 签名公钥可信根缺失、#8 部分唯一索引金仓兼容性未验证、#13 trip_physiological_snapshot 表结构缺失）、6 个轻微（#9 #10 #11 #12 #14 #15）。

质询报告（b_v4_challenge_v2.md）结论为 LOCATED，质询确认了诊断报告的问题定位正确，仅指出整体质量评价段中一般/轻微计数对调的文字错误（不影响问题本身的判定）。组件B内部循环实际轮次 2，未耗尽最大轮次 12，提前完成且结论有效。

依据判定标准：审查报告包含严重及一般等级的问题，判定为 RETRY。

## 需要解决的问题（仅 RETRY 时存在）

- **问题描述**：IoTDA 上行消息消费与 outbox 写入之间存在事务边界断裂——Kafka 偏移量提交与数据库 outbox 写入分属独立事务，outbox 写入失败 + 偏移量已提交时告警事件静默丢失
- **所在位置**：§3.5.1 IoTDA 上行消息消费适配器 + §3.3.1 云端事件总线实现
- **严重程度**：严重
- **改进建议**：采用手动提交偏移量模式（enable.auto.commit=false），仅 outbox 写入成功后才提交偏移量；或补充分布式事务协调方案

- **问题描述**：语音存证授权解密依赖云端 DEW 下发临时密钥，事故后车载终端离线导致解密链断裂，与需求 §六.2 三类授权场景的可用性要求矛盾
- **所在位置**：§3.6.2 授权解密段落 + §3.5.5 DEW 密钥管理
- **严重程度**：严重
- **改进建议**：增加离线解密路径（云端保存加密会话密钥副本、设备密钥包裹存储于 OBS）；增加现场解密路径（物理安全令牌 + DEW 预下发离线授权码）

- **问题描述**：OutboxRelayer 多实例部署下 SELECT 与 UPDATE 之间存在时间窗口，可能导致同一条记录被多个实例投递至 Kafka，设计声称的"保证被唯一实例获取"不成立
- **所在位置**：§6.3 并发策略表 + §4 场景 2 步骤 3
- **严重程度**：一般
- **改进建议**：将 SELECT 改为 SELECT ... FOR UPDATE SKIP LOCKED，或明确承认 at-least-once 语义包含投递层面重复

- **问题描述**：fleet_dashboard_projection 事件驱动即时增量聚合存在读-计算-写竞争窗口，并发告警事件可能导致增量聚合结果被覆盖丢失
- **所在位置**：§3.2.3 P2 缓存/物化视图策略路径 A
- **严重程度**：一般
- **改进建议**：在路径 A 描述中显式声明并发竞争取舍，注明路径 B 在下一个 5 分钟周期修正

- **问题描述**：脱敏校验门控的具体挂载点、组件归属、Schema 文件路径、校验失败后完整行为链未明确，可编码性不足
- **所在位置**：§3.6.1 脱敏校验门控段落
- **严重程度**：一般
- **改进建议**：补充门控组件归属、挂载方式（interceptor 或前置处理器）、Schema 路径、校验失败后完整行为链

- **问题描述**：家属 APP 告警通知存在 WebSocket / SMN / Push Kit 三条并行推送通道，APP 前台时可能收到重复通知，缺乏去重设计
- **所在位置**：§3.4.6 NotificationPort + §3.5.3 SMN 模板 + §3.7.2 部署图
- **严重程度**：一般
- **改进建议**：前台时仅走 WebSocket、后台时走 Push Kit，APP 侧基于 event_id 本地去重

- **问题描述**：OTA 固件签名验证所需的 RSA-2048 公钥来源、预置方式、防篡改保护未说明，缺少可信根设计
- **所在位置**：§3.4.5 车载终端侧 OTA 客户端
- **严重程度**：一般
- **改进建议**：公钥预烧录至 TEE/KeyStore 安全存储，或通过设备主密钥 HMAC 签名保护

- **问题描述**：guardianship 表部分唯一索引依赖 PostgreSQL 部分唯一索引语法，金仓数据库兼容性未经验证
- **所在位置**：§3.1.2 guardianship 关联表 + §6.3 并发策略
- **严重程度**：一般
- **改进建议**：补充金仓版本号及验证说明，给出金仓不支持时的悲观锁替代方案

- **问题描述**：trip_physiological_snapshot 表被三条独立数据路径使用，但完整列清单从未显式定义，三条路径的实现者可能做出不一致假设
- **所在位置**：§3.1.3 策略 B + §3.2.2a + §3.7.4
- **严重程度**：一般
- **改进建议**：以表格形式定义完整列结构，标注各数据来源对写入列的义务
