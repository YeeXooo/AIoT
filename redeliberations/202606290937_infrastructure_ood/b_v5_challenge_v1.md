### 1. 证据充分性

**[问题-严重]** 问题 1.2（DriverHealthProfile 独立查询不可行）的证据不充分。报告断言 `@OneToOne` + `@MapsId` 映射下该实体"无法由外层查询直接按健康指标条件独立检索"，但 JPA 体系中 `EntityManager.createQuery()` 或 `@Query` 注解可在任意实体上执行 JPQL/原生 SQL，完全独立于关系映射。报告未验证设计中是否已存在独立查询表达式（如声明"通过投影表或原生 SQL"），仅凭映射关系推断查询不可行，属于未验证假设。

**[建议]** 验证设计文档中是否已声明 DriverHealthProfile 的独立查询路径。若设计已说明"通过原生 SQL 查询"或"通过投影表查询"，则该问题不成立；若未声明，应将问题描述修正为"未显式给出独立查询实现路径"，而非断言"无法"。

**[问题-一般]** 问题 1.1（缺失概念级 DDL 概要）对需求"概念级 DDL 概要"的解释可能过严。需求 §一.5 原文为"概念级，不含完整 DDL，但需展示关键表的结构定义"——以 Markdown 表格列出字段名、类型、约束是否已满足"概念级展示"？报告将此解读为必须包含 `CREATE TABLE` 语法形式，但需求原文的"概念级"与"不含完整 DDL"暗示对语法形式的要求是宽松的。报告未论证表格形式为何不满足需求。

**[建议]** 明确判断标准：若需求方期望的是 SQL 语法级别的 DDL 模板，则报告成立；若表格形式已可接受，则需调整问题定性为"可进一步加强"而非"缺失"。

**[问题-严重]** 任务明确要求从"接口定义是否足以支持下游消费者"视角评估，但审查报告仅在问题 3.2 中涉及 VehicleStateBuffer 的方法签名。以下端口适配器的接口定义完备性未被评估：

- **OTADeliveryPort**：需求 §四.5 要求支持断点续传和传输进度回调。审查报告仅在 3.1 讨论了 OTA 安全风险，未评估端口方法契约（分片请求/响应、进度回调签名、续传参数）是否足以支持车载终端实现分片接收和断点续传。
- **DrivingBehaviorTrackingPort**：需求 §四.3 定义了 `onHardBrakingDetected` / `onHardAccelerationDetected` 方法契约，审查未评估其是否完整（阈值上下文、事件载荷字段、返回值语义）。
- **CameraOcclusionDetectionPort**：需求 §四.4 定义了遮挡检测回调，审查未涉及。
- **NotificationPort**：历史迭代第 2 轮第 9 条曾指出"端口方法契约未在设计中呈现"，审查报告未跟进此问题是否已在 v5 中修复。
- **RescueReportPort**：历史迭代第 2 轮第 11 条曾指出 HTTP API 备选路径缺乏接口规范，审查报告未跟进。
- **MediaSessionPort**：需求 §四.8 定义了会话创建/拆除方法，是否足以支撑 SparkRTC 房间生命周期管理（如会话超时、异常断线重连、并发会话数限制）未被评估。

**[建议]** 逐一审查所有外部端口适配器的接口定义，判断方法签名、参数类型、返回值约定、异常抛出声明是否足以被下游实现者（适配器编码者、端口调用方服务）直接使用。尤其是历史迭代中已暴露的问题点，应逐条确认修复状态。

**[问题-一般]** 任务要求评估"异常场景和边界条件是否已考虑"，审查报告仅覆盖了 OTA 安全中断（3.1）和 buffer 窗口竞争（3.4）两个边界场景。以下常见边界条件未被覆盖：

- Outbox 投递器在消息队列不可用时的行为（持续重试的磁盘膨胀、死信迁移时机）
- 分库分表场景下跨分片聚合查询的超时与降级策略
- 边缘侧 SQLite 数据库容量上限及触发清理策略
- SparkRTC 房间创建失败、会话中途异常断线的处理
- IoTDA MQTT 连接断开后边缘侧离线消息的本地存储上限
- DEW 密钥服务不可用时的语音存证加密/解密降级策略

**[建议]** 补充边界条件维度的系统评估，不要求穷举，但至少覆盖每个基础设施组件的主要异常路径。

## 质询要点

- **问题**：问题 1.2（DriverHealthProfile 独立查询不可行）基于 JPA 映射关系推断查询不可行，但 `@OneToOne` + `@MapsId` 不阻止通过 JPQL/原生 SQL 对该表独立查询，这一断言缺少实际验证。
- **原因**：若该判断错误，将引导设计作者增加不必要的索引或架构变更，浪费迭代资源并降低审查报告整体可信度。
- **建议方向**：确认设计文档中是否已有独立查询路径声明；若已有，撤销该问题；若没有，将问题修正为"缺少独立查询实现路径的显式说明"，而非"无法支持"。

- **问题**：审查报告未覆盖任务明确要求的"接口定义是否足以支持下游消费者"维度的系统评估，多个外部端口适配器（OTADeliveryPort、DrivingBehaviorTrackingPort、CameraOcclusionDetectionPort、NotificationPort、RescueReportPort、MediaSessionPort）的接口完备性未被审查。
- **原因**：接口定义是基础设施层设计的核心交付物，直接决定下游适配器编码者和端口调用方能否独立工作。遗漏此维度意味着审查报告在最关键的"可编码性"方面存在盲区。
- **建议方向**：在下一轮审查中为每个外部端口适配器增设接口完备性检查项，逐端口判断：方法签名是否完整、参数类型是否明确、返回值是否定义、异常路径是否声明。

- **问题**：边界条件覆盖不足，仅涉及 2 个场景（OTA 安全、buffer 竞争），遗漏 outbox 投递、分片查询、边缘存储、SparkRTC 断线、IoTDA 断连、DEW 不可用等关键异常路径。
- **原因**：基础设施层设计的关键价值之一是定义异常和边界行为。若审查报告未覆盖这些维度，设计文档中的边界缺陷将遗漏至编码阶段，增加返工成本。
- **建议方向**：为每个基础设施子系统（outbox、分片、边缘存储、SparkRTC、MQTT、密钥管理）逐项检查是否定义了主异常路径的处理策略。
