## 一、需求响应充分度

### 1.1 缺失需求 §一.5 要求的概念级 DDL 概要

| 属性 | 内容 |
|------|------|
| **问题描述** | 需求 §一.5 明确要求产出"金仓数据库的 CREATE TABLE DDL 概要（概念级，不含完整 DDL，但需展示关键表的结构定义）"。当前文档 §3.1.1-§3.1.2 仅以 Markdown 表格列出字段名和类型，未提供任何 DDL 语法级别的结构展示（如 `CREATE TABLE trip (...)` 形式的概要）。表格描述了列定义，但缺少表级约束（如 FOREIGN KEY 子句、CHECK 约束语法、索引的 CREATE INDEX 形式）的 SQL 表达，编码实现者无法直接从设计文档还原数据库 schema。 |
| **所在位置** | 需求 §一.5 vs 设计 §3.1.1-§3.1.3 |
| **严重程度** | 一般 |
| **改进建议** | 在 §3.1 末尾新增"概念级 DDL 概要"小节，以 `CREATE TABLE` 语法展示 1-2 张代表性表（如 `trip` 和 `safety_alert_event`），含完整列定义、PRIMARY KEY、FOREIGN KEY、UNIQUE、NOT NULL 等约束，以及代表性索引的 `CREATE INDEX` 语句。其余表可概述关键约束差异。 |

### 1.2 DriverHealthProfile 的表设计无法支持需求 §一.3 要求的独立查询

| 属性 | 内容 |
|------|------|
| **问题描述** | 需求 §一.3 明确指出 DriverHealthProfile（E-03）"不是聚合根但需要独立查询"——这意味着该实体应有不依赖 Driver 聚合加载的独立查询路径。但当前设计 §3.1.2 将其表的主键设为 `driver_id`，通过 JPA `@OneToOne` + `@MapsId` 共享 Driver 主键，索引仅 `(driver_id)`。此映射下，DriverHealthProfile 必须通过 Driver 聚合加载（级联），无法由外层查询直接按健康指标条件（如"查询所有血型为 X 的驾驶员"）独立检索，无法满足"需要独立查询"的要求。 |
| **所在位置** | 需求 §一.3 vs 设计 §3.1.2（DriverHealthProfile 表结构） |
| **严重程度** | 一般 |
| **改进建议** | 若独立查询需求确实存在（如按 `blood_type` 筛选高危驾驶员），应至少补充按 `blood_type` 或 `chronic_history` 等可查询字段的索引，并说明独立查询路径是通过 `EntityManager.createNativeQuery` 直接在 `driver_health_profile` 表上执行而非经过 JPA 关系映射加载。或在设计文档中明确声明独立查询仅通过投影表/原生 SQL 实现，不依赖 JPA 关系。 |

## 二、事实错误与逻辑矛盾

### 2.1 SystemAccountRepository.findByDriver 的 JPQL 在 §3.2.1 与 §3.2.2 中矛盾

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.2.1 的仓储实现总览表中，`SystemAccountRepository.findByDriver` 的 JPQL 写为 `JOIN a.guardianships g`——这隐含 `SystemAccountEntity` 上存在一个 JPA `@OneToMany` 映射字段 `guardianships`。但 §3.1.2 明确 `guardianship` 是独立的关联表（多对多关系），不属于 SystemAccount 聚合的内部字段。而 §3.2.2 的正确 JPQL 为 `JOIN guardianship g ON a.id = g.accountId`（跨表 JOIN）。两条 JPQL 对同一查询给出了互相矛盾的实现方式，其中 §3.2.1 的写法无法直接编译通过（除非编码者在 Entity 上额外添加 `guardianships` 字段，但这与设计意图矛盾）。 |
| **所在位置** | §3.2.1 第 408 行 vs §3.2.2 第 425 行 |
| **严重程度** | 严重 |
| **改进建议** | 将 §3.2.1 表中的 JPQL 修正为与 §3.2.2 一致的跨表 JOIN 写法：`@Query("SELECT a FROM SystemAccountEntity a, guardianship g WHERE a.id = g.accountId AND g.driverId = :driverId AND g.revokedAt IS NULL")`，或直接引用 §3.2.2 的写法。 |

## 三、深度与完整性

### 3.1 OTA 升级刷写期间安全监测中断的安全风险未被承认

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.4.5 描述 OTA 刷写阶段"车载终端暂停所有感知和判定业务（安全降级为仅保留 HMI 基本反馈），预计中断窗口 ≤2 分钟"。这意味着在固件刷写期间，碰撞失能检测、疲劳判定、路怒检测等全部安全功能停止运行，但设计未对"中断期间发生安全事件如何处理"给出任何陈述。这是一个系统性安全风险，需求文档和领域层均未授权 OTA 过程中可停机安全监测。作为基础设施层设计，至少应标注此安全风险并说明对应用层/领域层的约束要求（如：禁止在行驶中执行 OTA，或 OTA 仅允许在车辆驻车熄火状态下触发）。 |
| **所在位置** | §3.4.5 第 765 行 |
| **严重程度** | 一般 |
| **改进建议** | 补充 OTA 刷写前置条件声明：ODADeliveryPort 或 DS-15 应在触发刷写前校验车辆状态为驻车熄火（可通过 VehicleStateSnapshot 的 `doorLockStatus` 和 `speed` 判断），行驶中仅允许接收和暂存升级包、禁止进入刷写阶段。或明确标注此为已由需求方接受的已知风险并给出接受理由。 |

### 3.2 VehicleStateBuffer.getSnapshots 方法签名与需求中的端口契约不一致

| 属性 | 内容 |
|------|------|
| **问题描述** | 需求 §四.1 的端口方法契约为 `getSnapshotWindow(Instant from, Instant to): List<VehicleStateSnapshot>`，即方法名为 `getSnapshotWindow`，参数是两个时间点。但设计文档 §3.4.1 的回取接口描述为 `getSnapshots(tripId, window)`——方法名改为 `getSnapshots`，参数改为 `tripId` + `window`（窗口对象）。两者在方法签名层面不一致，直接编码实现时将产生接口不匹配。 |
| **所在位置** | 需求 §四.1 vs 设计 §3.4.1 |
| **严重程度** | 一般 |
| **改进建议** | 统一端口方法签名。若设计认为带 `tripId` 比纯时间窗口更有意义，应在修订记录中说明变更理由并与需求方确认。同样需检查 PhysiologicalDataBuffer §3.4.2 的 `getReadings(tripId, window)` 是否与需求中的 `getPhysiologicalWindow(Instant, Instant)` 一致。 |

### 3.3 guardianship 表广播表策略下的悲观锁替代方案未说明跨分片 SELECT...FOR UPDATE 的执行方式

| 属性 | 内容 |
|------|------|
| **问题描述** | §3.1.2 的"金仓不支持时的替代方案"中描述：在 guardianship 广播表上执行 `SELECT ... FOR UPDATE` 锁定活跃行。但 guardianship 是广播表（每个分片冗余一份，见 §3.5.2），所有行遍布每个分片。而 `SELECT ... FOR UPDATE` 在 ShardingSphere-JDBC 中对广播表的实际行为是：仅对路由命中的那个分片上的行加锁（通常仅一个分片），还是对所有分片广播加锁？文档未说明。如果仅锁定一个分片，另一分片上的并发写入仍可能绕过锁检查。 |
| **所在位置** | §3.1.2 第 266 行（替代方案段落）vs §3.5.2 广播表配置 |
| **严重程度** | 一般 |
| **改进建议** | 明确在 ShardingSphere-JDBC 广播表上执行 `SELECT ... FOR UPDATE` 的实际行为（是否广播至所有分片），或在替代方案中注明"广播表上 SELECT...FOR UPDATE 的行为需在编码阶段用例测试验证"，或改用非数据库层的分布式锁（如 Redis `SET NX`）作为广播表加锁方案。 |

### 3.4 边缘侧 EmergencyRescueService 的碰撞判定时间窗口与 ring buffer 容量之间的量化关系未给出

| 属性 | 内容 |
|------|------|
| **问题描述** | VehicleStateBuffer 回取窗口为 30s（§3.4.1），PhysiologicalDataBuffer 为 ≥10s（§3.4.2），但 DS-06 EmergencyResponseService 在碰撞判定时需要同时获取两个缓冲的数据构建 RescueReport。需求 §四.2 要求缓冲覆盖"碰撞前后"的时间窗口。两个缓冲的容量独立配置，无机制确保在碰撞发生后的安全观察窗口（§3.4.2a 默认 5s）期满时，两个缓冲均含有碰撞前完整的 30s + 10s 数据。若边缘侧在长行程中运行数小时，ring buffer 的已淘汰数据无法回取——这对碰撞前数据获取构成风险，但设计未说明。 |
| **所在位置** | §3.4.1、§3.4.2、§3.4.2a |
| **严重程度** | 轻微 |
| **改进建议** | 补充说明：碰撞前缓冲数据的可用性由 ring buffer 容量（固定 30s/10s）保证——EmergencyResponseService 必须在碰撞发生后的安全观察窗口内（≤5s）完成碰撞判定并调用 `getSnapshots`/`getReadings`，此时缓冲中必有碰撞前完整窗口数据。若判定延迟超过缓冲容量，则抛出 `BufferException.WindowNotCoveredException` 降级上报。 |

### 3.5 `domain_event_dlq` 表主键使用 BIGSERIAL 与文档声明的统一主键策略（应用层 UUID）不一致

| 属性 | 内容 |

| **所在位置** | 第 7 行（统一主键策略）vs §3.3.3 第 601 行（DLQ 表结构） |
