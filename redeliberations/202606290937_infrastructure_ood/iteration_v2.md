## 需要解决的问题

- **问题描述**：`fleet_dashboard_projection` 投影表缺少主键或唯一约束定义，导致数据同步策略中依赖的 `INSERT ... ON CONFLICT DO UPDATE` 操作无法执行
- **所在位置**：§3.2.3 投影表 P2 字段定义
- **严重程度**：严重
- **改进建议**：明确 P2 表的主键或唯一约束，建议以 `(fleet_id, risk_level, alert_type)` 作为复合唯一约束；在字段表中显式标注主键/唯一约束列

- **问题描述**：`trajectory_projection` 投影表的分片策略未明确，未纳入 §3.1.5 分片表清单，PK 为 BIGSERIAL 在跨分片环境下会产生主键冲突
- **所在位置**：§3.2.3 投影表 P3 vs §3.1.5 分片范围
- **严重程度**：一般
- **改进建议**：明确分片归属（建议按 `vehicle_id` 分片），PK 改为应用层生成 UUID 或复合分区键，补充到 §3.1.5 分片清单

- **问题描述**：心跳监控机制缺少数据支撑——Vehicle 表中无 `last_heartbeat_at` 字段，心跳监控定时任务无法判断超时
- **所在位置**：§3.1.1 Vehicle 表结构 vs §3.7.5 降级策略表
- **严重程度**：一般
- **改进建议**：在 Vehicle 表中增加 `last_heartbeat_at TIMESTAMP NULL` 列，说明心跳监控定时任务的执行逻辑和更新组件

- **问题描述**：Outbox 投递器轮询 SQL 查询条件存在逻辑缺陷——退避间隔为 per-event 变量，无法在单条 SQL 中动态计算
- **所在位置**：§3.3.2 索引设计 + §3.3.5 退避策略 + §4 场景 2 步骤 3
- **严重程度**：一般
- **改进建议**：明确实现方式（方案A：SQL 固定上界 60s + Java 层计算退避；方案 B：增加 `next_retry_at` 预计算列），同步修正三处描述的差异

- **问题描述**：`guardianship` 表权限变更策略存在内部矛盾——DELETE 与 REVOKED 状态标记两句话矛盾，且历史记录无法保留
- **所在位置**：§3.1.2 表结构 + §3.1.4 说明 + §6.3 并发策略
- **严重程度**：一般
- **改进建议**：统一表述，明确实际策略（推荐保留历史记录方案，将 PK 改为三列组合或增加自增主键），同步修正 §6.3 并发策略描述

- **问题描述**：IoTDA 上行消息消费机制不够具体，消费方式使用了"或"字未选定具体方案，无法指导编码实现
- **所在位置**：§3.5.1「IoTDA 上行消息消费适配器」小节
- **严重程度**：一般
- **改进建议**：选定一种具体方案（推荐 IoTDA 数据转发至 DMS Kafka），说明消费方式和消息体格式

- **问题描述**：`fleet_dashboard_projection` 缓存策略存在路径断裂——事件驱动的缓存失效仅删 Redis key，不更新投影表，导致"即时重算"实际回退到已延迟的投影表数据
- **所在位置**：§3.2.3 投影表 P2 缓存/物化视图策略
- **严重程度**：一般
- **改进建议**：明确"即时重算"含义（即时增量聚合并同步更新投影表，或接受 ≤5 分钟延迟并明确声明 SLO）

- **问题描述**：`@ElementCollection` 优化后集合被标记"已截断并冻结"，但未说明冻结的技术实现方式，JPA 脏检查可能触发级联 DELETE+INSERT 覆盖溢出数据
- **所在位置**：§3.1.3 策略 B 优化触发条件
- **严重程度**：一般
- **改进建议**：说明冻结的技术实现方式（不可变包装 / 清空引用 / 防御性标记），明确只读语义和异常行为

- **问题描述**：NotificationPort 适配器的端口方法契约未在设计中呈现，开发者无法从设计文档获知需实现的方法签名和返回值约定
- **所在位置**：§3.4.6「NotificationPort 适配器」
- **严重程度**：一般
- **改进建议**：在 §3.4.6 开头增加端口接口方法列表（告警推送、SOS 救援推送、绩效预警推送、家属常态快照推送方法契约概要）

- **问题描述**：OTA 升级的车载终端侧（接收端）设计缺失——分片接收、CRC32 校验、组装、刷写、回滚等边缘侧关键行为未涉及
- **所在位置**：§3.4.5 vs §3.7.1 边缘侧组件表
- **严重程度**：一般
- **改进建议**：新增"OTA 升级客户端（边缘侧）"小节，说明分片接收校验、组装刷写、回滚策略和断网续传逻辑

- **问题描述**：RescueReportPort 的 HTTP API 备选投递路径缺乏接口规范（endpoint URL、请求方法/格式、认证方式、可重试状态码）
- **所在位置**：§3.4.7「RescueReportPort 适配器」
- **严重程度**：一般
- **改进建议**：定义备用 HTTP API 接口规范概要，或标注为"外部依赖待确认"并给出占位约定

- **问题描述**：DEW 密钥轮转的离线车辆场景未覆盖——车辆离线期间未收到新密钥，重新上线后可能仍使用已标记为 DECRYPT_ONLY 的旧密钥加密新数据
- **所在位置**：§3.5.5「DEW 密钥管理与轮转策略」
- **严重程度**：一般
- **改进建议**：补充离线车辆密钥轮转处理策略（排队补推方案或宽限期方案），明确轮转过程中的密钥版本判定逻辑
