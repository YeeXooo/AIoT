## 当前审查结果

> 质询结论：**LOCATED**（诊断结论整体被确认可信）。下方第 2 条附带质询发现的论证偏差，请按修正后的口径处理。

### 问题 1（中等·认证链路不完整）：JWT refresh token 端点缺失
- **描述**：`LoginResponse`（§1.7）含 `refreshToken` 字段，§5.1 签发与校验流程第 5 步描述"JWT 过期前 5 分钟客户端可携带 refresh token 换取新 JWT"，但 §1.7 及整个 §1 未定义 `POST /api/v1/auth/refresh` 端点，致 JWT 刷新链路在 API 契约层面断裂，客户端只能等 token 过期后重新登录。
- **位置**：§1.7 Auth 端点表（行 713–716）vs §5.1 步骤 5（行 1882）
- **严重程度**：中等
- **建议**：在 §1.7 新增 `POST /api/v1/auth/refresh` 端点，定义 `RefreshTokenRequest`（含 `refreshToken: String`）与 `RefreshTokenResponse`（含新的 `accessToken`、`refreshToken`、`expiresIn`），参照现有 `POST /api/v1/auth/login` 端点风格。

### 问题 2（中等·错误响应不完整）：S5 和 S6 错误响应列表缺少基础错误码
- **描述**：S5（§1.5）错误响应仅列 `403`/`409`，缺 `400`（queryRescueHistory 查询参数无效）和 `404`（driverId/vehicleId 不存在）；S6（§1.6）仅列 `400`/`409`，缺 `404`（vehicleId 不存在于 queryUpgradeHistory）、`503`（IoTDA 下指令通道不可达）。
- **位置**：§1.5 错误响应（行 582–584）、§1.6 错误响应（行 703–706）
- **严重程度**：中等
- **建议**：在 §1.5 补充 `400`（查询参数无效）和 `404`（驾驶员/车辆不存在）；在 §1.6 补充 `404`（车辆/任务不存在）和 `503`（IoTDA 通道不可达，参照应用层 OOD §3.6 的 `AppError.IoTDAChannelFailure`）。
- **⚠ 质询修正（务必采纳）**：原诊断以 S3（401/403/409/503/504）作为"拥有完整基础错误码"的对比基线属事实偏差——S3 同样**不含** `400`/`404`。请改以 **S1（400/404/503）** 作为参考基线，去掉对 S3 的不实对照；并建议**一并将 S3 缺失的 `400`/`404` 基础错误码纳入本轮修复范围**，以保持错误码覆盖的一致性。

### 问题 3（一般·认证链路）：LoginRequest JSON 字段设计冗余
- **描述**：`LoginRequest`（§1.7）JSON 示例同时含 `credential`/`secret` 与 `phone`/`smsCode` 两组字段，独立的 `phone`/`smsCode`（null）与 `credential`/`secret` 负载复用形成语义冗余，且二者同时传入且值不同的后端行为未定义。
- **位置**：§1.7 LoginRequest JSON 示例（行 720–727）、枚举说明（行 730–732）
- **严重程度**：一般
- **建议**：二选一：(a) 移除 `phone`/`smsCode`，仅保留 `credential`+`secret`+`authMethod` 三字段，两模式共用同组字段（语义由 authMethod 决定）；(b) 若需区分，则 `PASSWORD` 用 `credential`+`secret`、`SMS_CODE` 用 `phone`+`smsCode`，避免两组字段同时出现于示例。

### 问题 4（一般·API 契约完整性）：错误响应体结构未定义
- **描述**：文档多处引用具体错误码（如 `BatchSizeExceeded`、`SecondaryAuthRequired` 等），但全文未定义 REST API 标准错误响应体格式；仅 §3.1 WebSocket 下行 `error` 隐式定义 `{ "code", "message" }`，未必适用于 REST 端点。应用层 OOD §6.1 已有完整 `AppError` 枚举，API OOD 未承接为 REST 错误响应契约。
- **位置**：§1 各节错误响应块（行 68–71、116–119、329–335、487–490、582–584、703–706）vs 应用层 OOD §6.1 AppError 枚举
- **严重程度**：一般
- **建议**：在 §一 总述段落后新增"**REST 错误响应体约定**"块（与 DELETE 响应码约定、401 统一处理约定并列），定义统一格式，建议为 `{ "errorCode": "BatchSizeExceeded", "message": "...", "requestId": "trace-uuid" }`，并明确 `errorCode` 映射到应用层 `AppError` 枚举变体名。

### 问题 5（一般·完整性）：S4 绩效预警订阅缺少取消订阅端点
- **描述**：S4 有 `POST /api/v1/fleet/performance-warning-subscription` 订阅端点，但无对应 `DELETE` 退订端点，订阅完整语义应含开通与关闭两方向；当前管理员一旦订阅无法主动退订。
- **位置**：§1.4 S4 端点表（行 350）vs S3 的 `POST/DELETE media-session` 对称设计
- **严重程度**：一般
- **建议**：在 §1.4 S4 端点表新增 `DELETE /api/v1/fleet/performance-warning-subscription/{subscriptionId}`，或通过 PUT 更新订阅状态（`{ "active": false }`）。

### 问题 6（轻微·安全设计不完整）：令牌桶容量参数未定义
- **描述**：§5.3 限流采用令牌桶算法，列出各层级速率限制并声明"突发容忍度为桶容量的 1.5 倍"，但未定义任何层级的桶容量（capacity），导致算法无法完整配置、"1.5 倍"比例失去基数意义。
- **位置**：§5.3 接口限流策略（行 1918–1932）
- **严重程度**：轻微
- **建议**：为各限流层级补充桶容量规格，如家属角色速率 30 req/s、桶容量 45；全局 1000 req/s、桶容量 1500。

### 问题 7（轻微·错误响应边界不一致）：S2 `404` 语义与其他服务不一致
- **描述**：S2（§1.2）声明 tripId 不存在时返回 `200 OK` + 空集合（不视为错误），而 S1/S4 对不存在资源返回 `404`，造成消费者需记忆两套判断规则。
- **位置**：§1.2 错误响应（行 117–119）vs §1.1（行 70）、§1.4（行 489）
- **严重程度**：轻微
- **建议**：二选一：(a) 统一为 `404`；(b) 在 S2 错误响应中补充设计理由说明（如"避免 404 向调用方泄露行程 ID 的存在性信息"）。
