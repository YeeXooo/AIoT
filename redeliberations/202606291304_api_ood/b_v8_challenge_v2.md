## 逐维度审查

### 1. 证据充分性

**[通过]** 问题1（windowPosition缺失）的事实认定经实际文件交叉验证——API OOD `a_v8_copy_from_v7.md` §1.3 行229 含 `windowPosition` 字段，应用层 OOD `docs/ood_application.md` §4.3 行606–610 `ControlVehicleWindowRequest` DTO 确实缺少该字段。证据链完整。

**[通过]** 问题4（secondaryAuthToken缺失）、问题6（RequestMediaSessionResponse字段缺失）的跨层不一致认定均经实际文件验证，应用层 OOD 中对应 DTO 确实缺少所述字段。

**[通过]** 问题3（createRescueReport缺失）经验证：应用层 OOD §3.5 接口方法表（行397–402）确实不包含 `createRescueReport` 方法，API OOD 中以"⚠ 接口契约待补"标注（行218）属实。

**[问题-严重]** §2–§5 各节独立审查的结论缺乏验证证据。修订说明（行251）对 §2 MQTT（"覆盖23个主题、QoS策略和Payload定义完备"）、§3 WebSocket（"协议完整"、"SparkRTC房间管理和高危豁免机制已定义"）、§4 ArkTS（"包含完整的REST调用清单、DTO类型定义"）、§5 安全设计（"覆盖JWT认证、令牌桶限流、X.509 MQTT鉴权、AES-256-GCM加密策略"）的审查结论均以一段概括性文字呈现。审查报告未提供逐项核对证据——例如：(a) §2 是否覆盖了 requirement.md 要求的三类 Topic 方向（边缘→云上报、云→边缘指令下发、云端推送→家属 APP），23 个主题是否逐个验证对应关系；(b) §3 是否分别覆盖了 WebSocket 信令协议和 SparkRTC 房间管理两个子项；(c) §5 的限流策略是否给出了具体速率和桶容量参数。审查报告将"我读了这些章节"与"我验证了这些章节满足需求"混为一谈。

**[建议]** 补充逐需求子项→设计产出段落的映射核对表，作为审查证据。例如：requirement.md §1 "S1流式判定会话管理" → API OOD §1.1 `POST /api/v1/risk/sessions` + `GET /api/v1/risk/sessions/{sessionId}`，逐项标注覆盖状态。

### 2. 逻辑完整性

**[通过]** 各问题的改进建议方向与问题描述一致，不存在内部矛盾。严重程度标注（v2 修正后）与总结表一致。

**[问题-严重]** 7/13 个问题（问题1/3/4/5/6/7/10）的事实认定均为"API OOD 中定义了字段 X，但应用层 OOD 同名 DTO 缺少字段 X"，改进建议均为"在 `docs/ood_application.md` 中补充字段 X"。这些问题的性质是**跨层依赖不一致**，而非 API OOD 自身的设计缺陷——API OOD 的字段定义本身是完整且正确的。将应用层 OOD 的缺陷作为 API OOD 质量审查报告的"问题"列出，模糊了"产出自身质量问题"与"外部依赖文档未同步"的边界。

特别需要注意的是问题6：API OOD §修订说明（v9）（行2023–2024）已由产出作者明确标注为"保留（无需修改本文件）——API OOD 的接口契约定义是完整的，问题根源在 `docs/ood_application.md`"。审查报告仍将其列为"严重"问题并重复相同建议。产出作者已完成认知闭环——识别了不一致、确认了本文件正确、指明了外部文档需修复——审查报告的重复列出不增加新信息，反而使产出作者接收到"需要修改的不是本文件"的冗余反馈。

**[建议]** 将跨层依赖不一致类问题从"API OOD 质量问题清单"中分离为独立的"跨层一致性清单"小节，标注为信息性条目而非缺陷。或至少为每个此类问题注明"API OOD 本文件接口定义正确，以下为应用层 OOD 需同步的字段清单"。

**[问题-中等]** 问题2（REST 层 publisher 死值）与问题13（ArkTS 层 SparkRTCRole 含 publisher）同一根源，拆分为两个独立问题。问题13 明确标注"此问题与问题 2（REST 层 publisher 死值）同源"。同一设计决策在 REST 层和 ArkTS 前端层的两个表现形式被计为两个独立问题，导致总结表统计数字不反映独立缺陷数。"轻微"列的 4 个问题中，若问题2 和问题13 均计入，则独立缺陷数实际为 3。

**[建议]** 将问题13 合并为问题2 的子条目（如"问题2a：REST 层"和"问题2b：ArkTS 层"），不独立计为两个问题编号。

**[问题-轻微]** 问题8 的版本标识判定与 API OOD 自身修订说明存在部分矛盾。API OOD §修订说明（v9）行2027 明确解释——标题已从"a_v8 / v8"修正为"a_v9 / v9"（与当前版本一致），文件名为系统指定路径。审查报告将此描述为"文档标题行（行1）为'a_v9 / v9'...但实际仅为第8轮"，其改进建议中的选项(b)（"在确认本版确为v9产出后统一"）复制了API OOD作者已完成的操作。该问题的标题标注并未考虑产出作者已有解释。

### 3. 覆盖完备性

**[通过]** 审查报告已覆盖 requirement.md 要求的五个部分（REST API、MQTT、WebSocket/SparkRTC、ArkTS 前端、安全设计），并对前序 7 轮迭代历史的遗留问题做了跟踪。

**[问题-严重]** requirement.md 明确要求的"需求响应充分度"未被执行式验证。需求文档列出各服务的子功能清单——S1（会话管理+历史查询，2项）、S2（干预查询+覆盖上报，2项）、S3（权限查询/管理+状态订阅+音视频对讲+车窗控制，4项）、S4（看板查询+钻取查询+报告生成/下载+绩效预警订阅，4项）、S5（SOS确认+救援授权管理+家属手动救援触发，3项）、S6（升级创建/查询+回滚指令下发+进度查询，3项）——共至少 18 个子需求。审查报告未逐一验证每个子需求是否有对应端点、端点设计是否完整。审查报告的检查停留在"该节存在"的粒度，而非"该节是否完整响应了需求的每个子项"。

**[建议]** 补充子需求→设计产出的映射核对清单，对每个子需求标注"已覆盖/部分覆盖/缺失"状态。

**[问题-严重]** requirement.md 约束条件"所有 API 端点应映射到应用层 OOD 中已定义的应用服务方法"的端到端覆盖率未验证。6 个服务共约 30+ 个端点，审查报告仅对少量端点（createRescueReport、cancelUpgradeTask、queryGuardianshipPermissions、issueSparkRTCToken）检查了映射关系，未给出完整端点→应用层方法的映射表和覆盖率统计。如果存在未映射的端点，这直接违反需求约束。

**[建议]** 补充"端点→应用层方法"完整映射清单（可简化为表格：端点路径 | 对应应用层方法 | 映射状态（已定义/预签名/缺失）），作为审查证据。

**[问题-中等]** 边界情况审查范围不完整。修订说明（行252）声称已检查"并发操作、幂等性、超时重试、降级策略等"并以 S5 verifyRescueToken（乐观锁并发）、S6 idempotencyKey（幂等）、车窗控制 504（超时）、WebSocket 重连退避（重试）为例。但这是示例性检查而非系统性检查——仅从 6 个服务中各挑 1 个端点做快照式验证，未覆盖所有高风险端点的完整错误处理矩阵（如 S3 window-control 的并发冲突窗口、S1 startRiskSession 的重复创建幂等性、S6 rollback 的级联状态不一致等）。

**[建议]** 补充"端点级错误处理矩阵"——至少对写操作端点逐一验证是否定义了并发、幂等、超时、降级的处理策略。

**[问题-中等]** requirement.md 约束条件"需考虑边缘—云协同架构的特殊性（边缘侧本地接口 vs 云端 REST API）"的审查深度不足。审查报告仅在修订说明中间接提及"§4 HMI 本地接口已覆盖"，未验证：(1) 边缘侧本地接口与云端 REST API 的职责边界是否在产出中明确划分（如哪些判定在边缘执行、哪些在云端执行）；(2) 边缘—云 MQTT 通道（§2）与云端 REST API（§1）是否构成端到端通信闭环。

## 质询要点（仅 CHALLENGED 时存在）

- **问题**：§2–§5 各节独立审查的结论缺乏验证证据，审查报告以概括性断言代替逐项核对。
- **原因**：任务描述明确要求侧重新一轮审查应覆盖的维度——"需求响应充分度、整体深度和完整性"。审查报告声称已完成 §2–§5 的独立审查，但未提供任何可被读者验证的证据（如需求子项→设计段落的映射表、每个子需求覆盖状态的标注），使得这些审查结论无法被独立验证，降低了审查报告的整体可信度。
- **建议方向**：(1) 对 requirement.md 的每个显式子需求（至少 S1–S6 各服务的子功能项、MQTT 三类 Topic、WebSocket 信令+SparkRTC、ArkTS 对接、安全四项机制）建立"需求→产出段落→覆盖状态"映射表；(2) 对约束条件（端点→应用层方法映射、边缘—云职责边界）逐项验证并提供覆盖率统计；(3) 对 §2–§5 的每项结论提供至少一行具体的核对记录（如"§2.1 Topic 路由表已覆盖需求要求的三类 Topic 方向：边缘→云感知上报对应 Topic x/y/z，云→边缘指令下发对应 Topic a/b/c，云端推送→家属 APP 对应 Topic d/e/f"）。

- **问题**：7/13 个问题指向应用层 OOD 而非 API OOD，问题归属错位。
- **原因**：审查报告将"API OOD 正确定义了字段而应用层 OOD 未同步"描述为 API OOD 的质量问题，混淆了"产出自身质量"与"跨层依赖一致性"两类不同性质的发现。从使用者视角（任务要求），若 API OOD 自身的接口契约定义完整且自洽，使用者可以直接依据 API OOD 进行实现，应用层 OOD 的不一致是其独立的质量问题。审查报告大量列入跨层依赖问题（占比 7/13），使得审查焦点偏离 API OOD 自身的缺陷。
- **建议方向**：(1) 将跨层依赖不一致类条目从"问题清单"中分离，作为独立的"跨层一致性说明"小节，标注为信息性条目；(2) 对已由产出作者显式处理的条目（如问题6），从问题清单中移除或降级为"已确认无需修改"的状态标注；(3) 将问题2 和问题13 合并为同根问题的两个子条目。

- **问题**：需求响应充分度和约束条件验证的缺失——子需求映射和端点-方法映射覆盖率未经系统检查。
- **原因**：requirement.md 列出的 18+ 个子需求项和"端点映射到应用层方法"的约束条件是衡量 API OOD 是否完整的关键标尺。审查报告未按此标尺逐项度量，导致无法回答"需求覆盖率达到多少"这一核心问题——而此问题正是任务描述"需求响应充分度"维度的核心。
- **建议方向**：(1) 构建"需求子项→设计产出"完整映射表并标注覆盖状态；(2) 构建"端点→应用层方法"完整映射表并标注映射状态（已定义/预签名/缺失）；(3) 在此基础上给出覆盖率统计和缺口清单。
