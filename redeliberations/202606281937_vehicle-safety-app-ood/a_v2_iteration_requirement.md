## 当前审查结果

本次审查（b_v1 / v1，质询结论：LOCATED）共发现 14 个问题，摘要如下：

### 高严重度问题（4 个）
1. **接口方法未提供具体签名**：六个应用服务的接口契约全部以自然语言散文描述，未提供具体方法名、参数类型名或返回值类型名。位于 §3.2–§3.7 各服务接口契约全部方法。改进建议：为每个方法提供具体签名，包含方法名、参数类型、返回值类型（如 `startMonitoringSession(driverId: DriverId, vehicleId: VehicleId): Result<SessionHandle, SessionError>`），对临时类型在应用层范围内给出字段定义或引用位置。

2. **DTO 定义完全缺失**：整个设计文档未定义任何一个 DTO（数据传输对象），下游接口层无法确定数据格式。位于全文缺失。改进建议：为每个应用服务的入口方法定义对应的输入 DTO 和输出 DTO，并说明 DTO 与领域对象之间的映射关系。

3. **关键类型未定义**：SensorReading、DriverStatusSnapshot、OverrideSignal 等处引用的类型未给出字段定义或结构说明。位于 §3.2 接口契约第 2 条、§3.4 接口契约第 1 条、§3.3 接口契约第 1 条。改进建议：为每个跨层传输类型给出字段级定义；对已在领域层定义的类型明确引用位置；对应用层特有的 DTO/VO 给出完整结构。

4. **异常类型仅有名称无定义**：§6.1、§6.2中列举了大量错误类型名称，但未定义任何错误枚举或错误码体系，每个错误携带什么结构化信息均未说明。位于 §6.1、§6.2。改进建议：定义应用层错误枚举（如 `enum AppError { SessionNotFound, PermissionDenied(reason), ... }`），并为每个错误类别定义携带的上下文信息。

### 中严重度问题（6 个）
5. **事务边界未定义**：应用层职责中包含"开启/提交/回滚事务"，但全文未对任何一条业务路径定义具体的事务边界。位于全文缺失。改进建议：为每个写操作类应用服务方法标注事务边界和隔离级别，特别对 S6 批量操作说明事务策略。

6. **S3 WebSocket 连接生命周期管理缺失**：未覆盖意外断开处理、重连恢复、心跳机制、连接数限制、离线消息补推等场景。位于 §3.4 接口契约（订阅/取消订阅）。改进建议：补充 WebSocket 连接生命周期管理策略，覆盖意外断开处理、重连状态恢复、心跳机制、连接限制、离线消息补推。

7. **S4 看板缓存无事件驱动失效机制**：高危告警最长可能 5 分钟延迟才反映在看板上，缓存未与领域事件联动。位于 §3.5 接口契约第 1 条、§7.2 并发策略。改进建议：明确看板缓存失效策略，如订阅高危告警事件时主动使对应缓存失效，或区分宏观统计缓存与高危车辆列表缓存。

8. **时序图存在跨层直接通信**：§5.2、§5.3 时序图中多处出现领域服务直接向应用服务/HMI 发消息，违反 §4.3 明确的设计原则。位于 §5.2（路径 2）、§5.3（路径 3）。改进建议：在时序图中显式引入事件总线作为事件路由中介，HMI 交互应经干预应用服务而非领域服务直控。

9. **S5 SOS 重试策略未具体化**：仅描述"按退避策略重试"，未定义退避算法、最大重试次数、最终失败处理方式。位于 §3.6 接口契约第 5 条。改进建议：明确退避算法（如指数退避 1s/2s/4s/8s）、最大重试次数（如 5 次）、最终失败处理（转人工干预 + 前端提示）。

10. **缺失验收测试场景规范**：设计目标声称"可验收导向"，但全文未提供任何验收测试场景或测试用例概要。位于全文缺失。改进建议：至少为 3 条核心时序图路径补充验收测试场景概要，覆盖正常路径和关键异常路径。

### 低严重度问题（4 个）
11. **时序图拼写错误**：§5.1 路径 1 时序图第 442 行 `FatgueDeterminationService` 应为 `FatigueDeterminationService`。改进建议：修正拼写。

12. **OTA 升级期间被抑制的 CAN 干预在回滚后的恢复策略未说明**：若在升级进行中触发回滚，CAN 干预能力何时恢复未明确。位于 §4.2 链路 D。改进建议：明确回滚过程中的 CAN 干预恢复时机。

13. **S3 订阅推送中"离线"状态的完整语义未定义**：未区分"无活跃行程（Inactive）"和"连接中断（Disconnected）"两种状态。位于 §3.4 异常处理策略第 4 条。改进建议：区分两种状态并说明各自对应的 UI 展示语义。

14. **仓颉语言 interface 能力未利用**：以"不存在多态替换场景"为由统一使用 class，但六份接口契约完全可用仓颉 interface 先行定义契约、再由 class 实现。位于 §3.1 决策 A1。改进建议：考虑以仓颉 interface 定义六个应用服务的契约（仅声明方法签名），由 class 实现，并在 §2.2 补充仓颉中依赖注入的具体方式。
