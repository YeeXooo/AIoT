# OOD 设计方案审查报告（v1）

## 审查结果

REJECTED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 设计方案中六项应用服务全部以 `class` 建模，仓颉语言原生支持 `class` 类型，`class` 可持有对其它对象（领域服务、仓储、端口）的引用，适合作为编排者角色。

**[通过]** 设计中使用的 `Result<T, E>` 和 `Option<T>`（`Option.None`）类型在仓颉标准库中均有直接对应，无需自定义。

**[通过]** 设计未涉及需要仓颉不支持的类型形态（如多重继承、交叉类型等）。`class` 之间通过持有引用（而非继承）形成协作关系，符合仓颉类型系统约束。

**[轻微]** 设计方案 §3.1 决策 A1 中提到若未来需要接口抽象可抽取策略对象，当前暂不引入 `interface`。此决策在当下可行，但注意应用服务完全无接口抽象会使其在单元测试中不可 mock / 不可替换，未来如确有测试隔离需求，可考虑为每个应用服务提取最小接口。

### 2. 标准库与生态覆盖

**[通过]** 设计中依赖的集合类型、基础数据结构均在仓颉标准库覆盖范围内。

**[通过]** 设计中涉及的外部基础设施（MQTT/IoTDA、SparkRTC、SMN、120 救援中心等）均通过领域端口（Port）抽象隔离在应用层之下，实际实现归属基础设施层，不构成对仓颉标准库的依赖假设。

**[通过]** 领域事件总线（EventBus）的实现虽不在仓颉标准库中，但应用层仅负责订阅事件（不负责实现事件总线），事件总线的实现归属基础设施层，不阻塞应用层设计。

**[轻微]** 设计中多处提到 CQRS 读模型投影和 outbox 模式，其中 outbox 需要数据库支持事务性 outbox 表。这些属于基础设施层关注点，应用层设计方案中作为假设引用是合理的，但建议在设计文档中明确标注「事件总线/outbox 的基础设施实现假设」，以便后续基础设施层设计时有明确约束。

### 3. 语言特性可行性

**[通过]** 错误处理策略统一采用 `Result<T, E>` 模式（§六），仓颉原生支持 `Result` 类型及模式匹配，与设计方案一致。

**[通过]** 并发设计（§七）：应用服务设计为无状态，云端可水平扩展；边缘侧 RiskMonitoringService 运行于单线程环境、按时间序列顺序处理感知数据。该并发模型与仓颉的 fiber 并发模型兼容，无冲突。

**[通过]** 模块/包命名采用 `application.risk`、`domain.risk`、`domain.event`、`domain.model` 等点分层次结构，与仓颉 cjpm 项目组织方式兼容。

**[通过]** 设计中涉及的乐观锁（§七.2）和会话上下文管理（RiskMonitoringSession）均为设计级描述，仓颉可通过 `class` 成员变量和原子操作实现，语言特性无阻碍。

**[轻微]** 设计方案 §六中提到 C 类系统故障「抛异常」，但当前仓颉语言以 `Result<T, E>` 为主要错误处理手段，异常抛出（`throw`/`try-catch`）为补充机制。建议在错误处理策略中明确：C 类异常是由应用层捕获后转换为 `Result` 返回调用方，还是直接向上传播。当前描述中 S1~S6 各接口的异常处理均以返回错误/`Result` 为主，与「抛异常」的描述之间存在轻微不一致，建议统一。

### 4. 设计一致性

**[一般]** 服务间协作关系图（§4.1）中存在 S2 InterventionService → S6 OTAManagementService 的依赖箭头，但整个 §4.2「协作链路分解」中无任何文字描述该依赖关系。当前设计方案仅说明了三条协作链路（A：判定→干预、B：告警→推送、C：家属手动救援联动），均不涉及 S2 与 S6 之间的协作。该依赖箭头的存在缺乏依据——若该依赖为设计意图（例如 OTA 升级期间需抑制干预执行），则应在协作链路分解中补充说明；若为绘制疏漏，则应从图中移除。无论何种情况，当前文档不一致。

**[轻微]** 模块划分表（§2.1）中 S1 RiskMonitoringService 标注的依赖领域层模块为 `domain.risk`、`domain.event`、`domain.model`，但 S1 的详细协作关系描述（§3.2）中明确依赖 `LifeDetectionService`（活体遗留判定）和 `EmergencyResponseService`（碰撞失能判定），这两个领域服务按命名惯例应归属 `domain.life` 或 `domain.emergency` 等模块，未在 §2.1 表中体现。虽然 §2.2 已说明应用服务可依赖多个领域模块，但汇总表与详细描述之间的不一致建议修正，避免后续实现阶段误解模块编译依赖范围。

**[通过]** 各应用服务的职责描述清晰，接口契约（方法意图、参数/返回值语义、异常处理策略）完整，足以指导后续详细设计。

**[通过]** 模块间依赖方向为应用层单向依赖领域层，领域层不反向依赖应用层，符合 DDD 分层原则。

**[通过]** 跨功能域协作全部通过领域事件解耦（§4.3），无编译期循环依赖风险。

**[通过]** 所需覆盖的三条关键路径时序图（§五）均以 Mermaid sequenceDiagram 语法完整描述，涵盖了边缘端→云端→APP/大屏全链路。

### 5. 设计质量

**[通过]** 六个应用服务各自对应明确的功能域，职责划分清晰，遵循单一职责原则。

**[通过]** 应用层严格遵循「薄层」原则——不包含业务逻辑，仅负责编排、事务管理、安全门控和 DTO 转换。业务规则归属领域层，分层边界清晰。

**[通过]** 安全门控统一原则（§六.3）明确——认证、角色校验、二次身份验证、权限校验四步链式校验在应用层完成，任一步不通过即拒绝，避免「先放行再校验」的安全漏洞。

**[通过]** 设计决策（§八）均有明确的理由说明（A1~A5），设计意图可追溯。

**[通过]** 应用服务无状态设计使其便于水平扩展和单元测试，通过依赖注入可 mock 仓储和领域服务。

**[轻微]** S1 RiskMonitoringService 的职责涵盖边缘侧会话管理和云端查询 API 两类不同部署环境的关注点。虽然二者属于同一功能域，但建议在设计文档中明确标注边缘侧与云端侧的职责切分边界，以便后续实现时更清晰地指导模块拆分。

**[轻微]** 方案 §七.2 并发场景策略中提到「Vehicle 聚合乐观锁」、「SystemAccount 聚合乐观锁」，但两个聚合根的定义在领域层 OOD（a_v10_design_v1.md）中，应用层设计方案未引用领域层设计中对应聚合根的版本号或标识，建议在跨文档引用处添加指向领域层产出的明确路径，以增强可追溯性。

## 修改要求（REJECTED 时存在）

- **问题**：服务间协作关系图（§4.1）中存在 S2 InterventionService → S6 OTAManagementService 的依赖箭头，但 §4.2 协作链路分解中无任何文字描述该依赖关系，文档内部不一致。
- **原因**：该未解释的依赖关系导致设计的协作模型不完整——实现者无法判断 S2 与 S6 之间是否实际存在协作、协作通过何种机制（事件订阅？直接调用违反 §4.3 原则？）、以及该依赖的存在是否有业务理由。这属于设计缺陷需补充说明或修正。
- **建议方向**：在 §4.2 中补充 S2 与 S6 之间的协作链路说明（若确存在）；或从 §4.1 协作图中移除该箭头（若为绘制疏漏）。若 S2→S6 的协作确实存在，应明确是通过哪个领域事件实现解耦（如 OTA 升级完成事件触发干预状态查询刷新等）。
